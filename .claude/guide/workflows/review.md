# 验收反馈流程

> **如何测试功能、反馈问题、迭代完善**

---

## 🎯 流程概览

```
功能完成 → 你测试 → 发现问题 → 我分析 → 我修复 → 你再测试
```

---

## 📋 步骤1：功能完成

### 我会这样做

```
【功能完成】
✅ XX功能已完成

【效果说明】
现在用户可以：
1. [功能点1]
2. [功能点2]
3. [功能点3]

【测试指引】
你可以试试：
- [测试建议1]
- [测试建议2]
- [测试建议3]

【等待验收】
请测试一下，有任何问题告诉我。
```

---

## 📋 步骤2：你测试功能

### 测试时

**你可以**：
- ✅ 按照测试指引尝试功能
- ✅ 自由探索各种操作
- ✅ 尝试边界情况（极端输入、异常操作）
- ✅ 从用户角度体验流程

**不需要**：
- ❌ 不需要懂技术
- ❌ 不需要看代码
- ❌ 不需要担心"这样改会不会很麻烦"

---

## 📋 步骤3：发现问题

### 推荐格式

```
【问题类型】功能错误 / 体验不好 / 需求理解有误
【场景描述】我点击XX按钮后...
【期望效果】我希望...
【实际效果】现在的情况是...
```

### 示例1：功能错误

```
【问题类型】功能错误
【场景描述】我提交评分后
【期望效果】应该看到"提交成功"提示
【实际效果】页面没有任何反应，不知道是否提交成功
```

### 示例2：体验不好

```
【问题类型】体验不好
【场景描述】我提交游戏后
【期望效果】提交后能留在当前页面，再检查一下
【实际效果】现在直接跳转到列表页，我还没看清楚就跳走了
```

### 示例3：需求理解有误

```
【问题类型】需求理解有误
【场景描述】用户浏览游戏列表时
【期望效果】希望只看自己参与的游戏
【实际效果】现在显示了所有游戏，还要自己找
```

### 💡 不会描述？

直接说你的感受：

```
我觉得这个地方有点怪...
我不知道怎么操作...
我不太清楚这个按钮是干嘛的...
我觉得这个流程有点复杂...
```

**我会**：
- 追问细节
- 理解你的真实需求
- 提供解决方案

---

## 📋 步骤4：我分析问题

### 我会这样做

1. **复述确认问题**
   ```
   【问题确认】
   我理解了：你希望[复述问题]
   ```

2. **定位问题原因**（用你能理解的方式）
   ```
   【问题原因】
   现在的逻辑是：[当前逻辑]
   这就像[生活中的类比]，所以会出现这个问题
   ```

3. **提供解决方案**
   ```
   【解决方案】
   我建议改成：[新方案]
   这样就能解决[具体效果]
   ```

4. **等待你确认**
   ```
   【需要确认】
   这样改可以吗？还是你有其他想法？
   ```

### 示例

```
【问题确认】
我理解了：你希望提交游戏后不要直接跳走，
而是给你一个友好的提示，让你知道已经成功了。

【问题原因】
现在的逻辑是：提交成功 → 立即跳转
这就像你寄快递，前台直接把快递单拿走，不给你回执，
你会想"到底有没有寄成功？"

【解决方案】
我建议改成：
提交成功 → 显示提示框"提交成功！" → 3秒后自动跳转
（你也可以手动关闭提示框，多看一会儿）

【需要确认】
这样改可以吗？还是你有其他想法？
```

---

## 📋 步骤5：我修复问题

### 你确认方案后

**我会**：
1. ✅ 实施修改
2. ✅ 测试修改是否生效
3. ✅ 告诉你修改完成
4. ✅ 请你再次验收

**示例**:
```
【修复完成】
✅ 问题已修复

【修改说明】
现在提交游戏后会：
1. 显示"提交成功！"提示框（绿色背景）
2. 3秒后自动跳转到列表页
3. 你也可以点击"关闭"按钮，手动关闭提示

【请再次测试】
请试试现在的效果，看看是否满意？
```

---

## 📋 步骤6：迭代完善

### 如果还有问题

**重复流程**：
```
你测试 → 发现新问题 → 我分析 → 我修复 → 你再测试
```

**直到**：
- ✅ 你完全满意
- ✅ 或达到合理方案（我会解释限制）

### 如果满意

```
你: 好的，这样就可以了
我: ✅ 完成！
```

我会：
1. ✅ 更新文档
2. ✅ 标记任务完成
3. ✅ 准备下一个任务

## 💡 反馈技巧

### 对产品经理（蜡烛先生）

**好的反馈**：
- ✅ 描述具体场景（"我点击XX后..."）
- ✅ 说明期望效果（"我希望..."）
- ✅ 诚实表达感受（"我觉得有点怪..."）
- ✅ 不确定时就说"不确定"

**可以**：
- ✅ 随时提出修改意见
- ✅ 要求多次调整
- ✅ 改变主意（在开发早期）
- ✅ 说"我不喜欢，但我不知道怎么改"

### 对技术开发（Claude）

**好的处理**：
- ✅ 复述确认问题
- ✅ 用类比说明原因
- ✅ 提供完整解决方案
- ✅ 耐心解释技术限制
- ✅ 诚实说明可行性

**避免**：
- ❌ 不要说"这很难"
- ❌ 不要说"你不懂"
- ❌ 不要敷衍了事
- ❌ 不要让产品经理做技术决策

## 🎯 常见问题

### Q：我觉得不对，但说不出哪里不对

直接说：
```
我觉得这个地方不太对，但是说不上来...
```

**我会**：
- 引导你描述感受
- 提供多个选项
- 或给出建议方案

### Q：改了很多次还是不满意

**我会**：
- 🤔 重新理解需求
- 📊 提供完全不同的方案
- 💡 询问核心诉求

**你可以**：
- 🎨 描述理想效果（"我最希望的是..."）
- 🖼️ 举例类比（"就像XX那样..."）
- ❓ 问我建议（"你觉得怎么改更好？"）

### Q：这个改起来很麻烦吗？

**不需要担心这个问题**！

- ✅ 改动小时，我直接改
- ✅ 改动大时，我会告诉你影响
- ✅ 你只需要关注"这是不是你想要的"

**我不会问**："这个改起来很麻烦，要不要改？"

**我会说**："这个需要改动较大，预计需要2小时，你觉得值得吗？"

## 📊 问题类型

### 类型1：功能错误（高优先级）

**特征**：
- 功能不工作
- 结果明显错误
- 影响核心流程

**处理**：
- ✅ 立即修复
- ✅ 优先级最高
- ✅ 不拖延

### 类型2：体验不好（中优先级）

**特征**：
- 功能能用但不顺手
- 流程有点复杂
- 提示不够友好

**处理**：
- ✅ 记录下来
- ✅ 评估改进成本
- ✅ 合理时优化

### 类型3：需求理解有误（高优先级）

**特征**：
- 做出来的不是想要的
- 理解有偏差
- 缺少关键功能

**处理**：
- ✅ 重新确认需求
- ✅ 调整方案
- ✅ 重新开发

### 好的验收反馈

- ✅ 问题描述清楚
- ✅ 双方理解一致
- ✅ 方案明确可行
- ✅ 最终效果满意

- ❌ 问题说不清楚
- ✅ 理解有偏差
- ❌ 反馈不及时
- ❌ 将就不满意
