# 惊蛰计划 - 协作工作流程

> 本文档说明蜡烛先生（产品经理）与 Claude（技术开发）的协作方式

---

## 🔄 总体流程

```
产品经理（蜡烛先生）          技术开发（Claude）
    ↓                            ↓
提出需求/想法         ←→   厘清细节/确认理解
    ↓                            ↓
确认原型/方案              出具技术方案
    ↓                            ↓
    双方达成一致，进入开发
    ↓                            ↓
                        执行开发任务
    ↓                            ↓
展示进度/成果         ←→   验收/反馈
    ↓                            ↓
    修改/完善                    ↓
        进入下一轮功能
```

---

## 📋 阶段1：需求确认

### 你的角色（产品经理）

**提出需求时，可以参考这个格式**（不是强制的）：

```
【需求类型】新功能 / 修改 / 问题修复
【场景描述】在什么情况下...
【期望效果】我希望...
【优先级】高 / 中 / 低（可选）
```

**示例**：
```
【需求类型】新功能
【场景描述】参赛者提交游戏后
【期望效果】应该能看到自己的游戏在列表中，并且能看到有多少人评分
【优先级】高
```

### 我的角色（技术开发）

**我会这样做**：
1. 确认需求类型（新功能/修改/问题）
2. 用产品经理能理解的方式复述确认
3. 提供技术方案说明（用"人话"解释）
4. 等待你确认

**示例回复**：
```
【需求确认】
我理解你的需求是：
参赛者提交游戏后，在游戏列表中能看到自己的游戏，
并且显示有多少人给这个游戏评过分。

【技术方案】
我打算这样做：
1. 在提交成功后，跳转到游戏列表页
2. 用一个特殊的标记突出显示自己的游戏
3. 显示评分人数（例如：12人已评分）

【你的理解】
这就像你在电商网站买东西后，订单列表会显示你的订单状态。

【需要确认】
这样理解对吗？你同意这个方案吗？
```

## 🛠️ 阶段2：开发实现

### 我的角色（技术开发）

**开发过程中**：
- 使用 TodoWrite 工具追踪所有任务
- 每完成一个模块，向你展示进度
- 定期汇报（频率由你决定）

**进度汇报格式**：
```
【今日进度】
✅ 已完成：用户注册功能
🔄 进行中：游戏提交功能（50%）
⏳ 待开始：评分系统

【遇到问题】
❓ 技术选型需要你确认：前端框架用React还是Vue？

【下一步】
继续开发游戏提交功能，预计今天完成。
```

### 你的角色（产品经理）

**你可以随时**：
- 📋 查看当前任务清单
- 📊 要求查看进度
- 🔄 提出修改意见
- ⏸️ 叫停某个功能
- ➕ 添加新需求

**关键原则**：
- 📌 **你看不懂代码没关系**，我会说明"改了什么效果"
- 📌 **你随时可以调整方向**
- 📌 **不必担心打扰我**，有问题随时问

---

## 🧪 阶段3：测试与迭代

### 你的角色（产品经理）

**测试时发现问题时**：

```
【问题类型】功能错误 / 体验不好 / 需求理解有误
【场景描述】我点击XX按钮后...
【期望效果】我希望...
【实际效果】现在的情况是...
```

**示例**：
```
【问题类型】体验不好
【场景描述】我提交游戏后，页面直接跳走了，但我还想再检查一下
【期望效果】提交后能留在当前页面，或者给我一个提示
【实际效果】现在直接跳转到列表页，我有点懵
```

### 我的角色（技术开发）

**我会这样处理**：
1. 复述确认问题
2. 定位问题原因（用你能理解的方式解释）
3. 提供解决方案
4. 实施修改
5. 请你再次验收

**示例回复**：
```
【问题确认】
我理解了：你希望提交游戏后不要直接跳走，
而是给你一个友好的提示，让你知道已经成功了。

【问题原因】
现在的逻辑是：提交成功 → 立即跳转
这就像你寄快递，前台直接把快递单拿走，不给你回执

【解决方案】
我建议改成：提交成功 → 弹出提示"提交成功" → 3秒后自动跳转
（你也可以手动关闭提示）

【需要确认】
这样改可以吗？还是你有其他想法？
```

## 📞 沟通机制

### 汇报频率

**我建议**：
- ✅ 每完成一个重要功能就告诉你
- ✅ 每天结束时总结今日进度
- ✅ 遇到需要决策的问题立即问你

**你可以随时**：
- 要求我更频繁/更少地汇报
- 主动询问进度

### 决策机制

**遇到技术问题时**：

❌ **我不会这样**：
- "我用React还是Vue？"

✅ **而会这样**：
- "我建议用X方案，原因是... 对你来说的好处是... 你觉得可以吗？"

**只在必要时才让你选择**，其他的我来做。

---

## 🗣️ 沟通风格

### 我的原则

1. **用中文沟通**
2. **用产品经理能理解的方式解释技术**
   - 多用类比
   - 少用技术术语
   - 必须用术语时会解释

**示例**：
```
❌ 不好的解释：
"我们需要用RESTful API，前端通过Ajax请求后端接口..."

✅ 好的解释：
"前端就像餐厅前台，后端就像厨房。
用户点菜后，前台把单子送到厨房，厨房做好后再送出来。
我们现在要设计的就是这个'传单'的规则。"
```

3. **诚实透明**
   - 不确定的会说"不确定"
   - 需要查资料会说"我查一下"
   - 做不到的会直接说"这有困难"

## 🎯 工作原则

### 我对你的承诺

1. ✅ **不擅自做主**：重要决策先问你
2. ✅ **不过度设计**：代码简洁够用就好
3. ✅ **及时汇报**：让你知道进度
4. ✅ **诚实透明**：不确定就是不确定

### 我希望你做的

1. ✅ **需求说清楚**：不确定时我会追问
2. ✅ **及时反馈**：看到问题就告诉我
3. ✅ **大胆决策**：不知道怎么选时，听我的建议
4. ✅ **保持沟通**：有任何想法随时说

## 📝 常见场景速查

| 场景 | 你怎么做 | 我怎么做 |
|-----|---------|---------|
| 提新需求 | 描述你想要什么 | 确认理解 → 给方案 → 等你确认 |
| 问进度 | "现在做到哪了？" | 告诉你当前状态和已完成事项 |
| 发现问题 | 描述问题场景 | 确认问题 → 找原因 → 给解决方案 |
| 想要修改 | "我觉得XX需要改一下" | 确认修改点 → 评估影响 → 实施修改 |
| 不确定需求 | "我不确定该怎么弄" | 提供选项让你选，或听我的建议 |
| 看不懂代码 | 不需要看代码 | 我用"人话"解释效果 |

## 🔄 工作流程总结

```
┌─────────────────────────────────────┐
│   你：用自然语言表达需求              │
│   我：确认理解，用产品经理语言反馈    │
│   你：确认方案                       │
│   我：执行开发，定期汇报进度          │
│   你：验收成果，提出修改意见          │
│   我：迭代完善                       │
└─────────────────────────────────────┘

核心理念：
- 你负责"做什么"（产品功能）
- 我负责"怎么做"（技术实现）
- 中间充分沟通确认
```
