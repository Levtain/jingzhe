## 🔴 P0优先级 - 必须修复 (2个)

### ✅ Q1: 吻合度公式的数学缺陷 - 已确认

**问题描述**:

当前公式 `吻合度 = 1 - |展示值 - 心理价| / 心理价` 存在严重的数学问题。

**具体场景**:

**场景1: 心理价1金币时的问题**
```
心理价: 1金币
- 实际0金币 → 吻合度 = 0%
- 实际2金币 → 吻合度 = 0% (|2-1|/1 = 1, 1-1=0)

问题:
- 心理价1金币时，任何偏差都导致吻合度骤降
- 不合理！偏差1金币和偏差10金币，惩罚一样
```

**场景2: 心理价20金币时的问题**
```
心理价: 20金币
- 实际10金币 → 吻合度 = 50%
- 实际30金币 → 吻合度 = 50%

问题:
- 心理价20金币时，即使差10金币，吻合度仍有50%
- 公平吗？
```

**数学缺陷**:
- 吻合度对心理价**敏感度不一致**
- 心理价越小，偏差影响越大
- 心理价越大，偏差影响越小

**选项**:

**A. 绝对偏差阈值（推荐）**
```
吻合度 = max(0, 1 - |展示值 - 心理价| / 5)
```
- 优点: 所有心理价的偏差影响一致
- 优点: 简单直观
- 缺点: 需要确定阈值5是否合理

**B. 相对+绝对混合**
```
吻合度 = (相对偏差 + 绝对偏差) / 2
相对偏差 = 1 - |展示值 - 心理价| / 心理价
绝对偏差 = 1 - |展示值 - 心理价| / 10
```
- 优点: 兼顾相对和绝对偏差
- 优点: 更平滑的衰减曲线
- 缺点: 计算复杂

**C. 分段线性**
```
if |展示值 - 心理价| <= 1:
    吻合度 = 100%
elif |展示值 - 心理价| <= 3:
    吻合度 = 80%
elif |展示值 - 心理价| <= 5:
    吻合度 = 60%
else:
    吻合度 = max(0, 1 - |展示值 - 心理价| / 10)
```
- 优点: 直观易懂
- 优点: 符合用户心理预期
- 缺点: 分段硬编码，不够灵活

**我的建议**: **选项A（绝对偏差阈值）**
- 理由: 公平性最好，所有心理价同等对待
- 阈值可以根据实际情况调整（5或10）

---

**✅ 用户确认**: 选项A（阈值调整为10）

**最终公式**:
```
吻合度 = max(0, 100 - 10 × |展示值 - 心理价|) %
```

**确认理由**:
1. **心理价1-10全覆盖**: 差1金币就扣10%吻合度，线性简单，观众口算就能懂
2. **公式统一**: 不再出现"心理价越小越吃亏"的悬崖
3. **天然归一化**: 10是系统上限，无需额外magic number

**对比示例**:

| 心理价 | 展示值 | 旧吻合度 | 新吻合度 | 观感 |
|--------|--------|----------|----------|------|
| 1 | 2 | 0% | 90% | ✅ 合理 |
| 10 | 5 | 50% | 50% | ✅ 一致 |
| 5 | 8 | 40% | 70% | ✅ 线性平滑 |

**新公式特性**:
- 差0 → 100%
- 差1 → 90%
- 差2 → 80%
- ...
- 差10 → 0%
- 差≥10 → 0% (max函数保护)

### ✅ Q2: 激励扭曲风险 - 已确认

**问题描述**:

用户可能**故意填低心理价**以获得"被低估"标签和徽章。

**具体场景**:
```
真实场景:
- 创作者认为游戏值10金币
- 但填5金币（故意低估）
- 目的: 获得"被低估"标签
- 结果: 实际收益7金币 → 吻合度40% → 标签"被低估"📈
- 收益: 获得"黑马发现者"徽章，社交媒体炫耀

动机分析:
- "被低估"标签 = 谦虚 + 作品受欢迎
- 社交媒体传播价值 > 准确的自我评估
```

**你的防滥用机制**:
```markdown
1. "过于谦虚"标签
2. 历史一致性检查
```

**我的担忧**:
- 这些机制可能**不够强**
- 用户可能**不在乎**"过于谦虚"标签（反而可能是荣誉？）
- 社交媒体的激励可能远大于系统内的约束

**选项**:

**A. 心理价位与历史收益挂钩（推荐）**
```
规则:
- 首次填写: 允许任意值（1-20金币）
- 后续填写: 不能偏离历史平均收益超过±50%
- 示例: 历史平均10金币，新游戏心理价必须在5-15金币范围内
```
- 优点: 强制约束，防止极端值
- 优点: 基于真实数据，更合理
- 缺点: 首次用户无历史数据

**B. "被低估"标签稀释**
```
规则:
- 如果连续3次"被低估" → 提示"您可能低估了自己的作品"
- 限制"被低估"徽章获取: 需要10次"被低估"且非连续
```
- 优点: 不强制限制用户自由
- 优点: 通过提示引导用户
- 缺点: 约束力较弱

**C. 引入"定价信誉分"**
```
规则:
- 初始100分
- 每次"过于谦虚": -5分
- 每次"过于自信": -3分
- 信誉分<80: 禁止获取徽章
```
- 优点: 量化约束，清晰透明
- 优点: 可以追溯历史行为
- 缺点: 增加系统复杂度

**我的建议**: **选项A（与历史收益挂钩）**
- 理由: 最强约束，从根本上防止滥用
- 首次用户可以给予一次"试错"机会

---

**✅ 用户确认**: 首届简化方案

**最终规则**:
```
心理价位填写与修改规则:
- 评分阶段开始前: 可以随意更改
- 评分阶段开始后: 非特殊情况不可更改
- 参考: 与"更改游戏链接"规则相同
```

**设计理念**:
- **首届简化**: 不考虑复杂的防滥用机制
- **灵活学习**: 允许用户在评分开始前调整预期
- **严肃对待**: 评分开始后锁定，避免频繁修改影响数据
- **特殊情况**: 保留管理员手动修改的权限（极端情况）

## 🟡 P1优先级 - 高优先级 (4个)

### ✅ Q3: 游戏类型的自然流量差异 - 已确认

**问题描述**:

不同游戏类型的打赏人数天然差异巨大，导致不公平。

**具体场景**:
```
解谜游戏: 50人打赏，平均3金币 → 总收益150金币
实验性游戏: 10人打赏，平均5金币 → 总收益50金币

截尾均值:
- 解谜: 3金币（去掉5个）
- 实验: 4金币（去掉1个）

问题:
- 解谜游戏作者: 心理价3金币，吻合度100% ✅
- 实验游戏作者: 心理价5金币，吻合度80% ⚠️

公平吗？实验游戏的"质量"可能更高（平均打赏5vs3）
```

**选项**:

**A. 引入打赏人数权重（推荐）**
```
加权吻合度 = 基础吻合度 × (1 + min(打赏人数/50, 0.2))
```
- 打赏50人: 吻合度×1.1（奖励10%）
- 打赏10人: 吻合度×1.0（不调整）
- 优点: 承认人气也是作品质量的一部分
- 缺点: 可能加剧"热门游戏越热门"的马太效应

**B. 分层展示**
```
- 样本充足版: 打赏≥30人 → 使用截尾均值
- 样本不足版: 打赏<30人 → 标记"样本量小"⚠️
```
- 优点: 明确告知数据可信度
- 缺点: 没有解决公平性问题

**C. 不处理**
```
保持现状，承认人气差异是自然现象
```
- 优点: 简单
- 缺点: 对小众类型不公平

**我的建议**: **选项A（打赏人数权重）**
- 理由: 承认人气也是作品质量的一部分
- 权重不宜过大（最多20%），避免过度激励

---

**✅ 用户确认**: 首届简化，后期优化

**最终规则**:
```
第一届处理方案:
- 不引入打赏人数权重
- 不区分游戏类型差异
- 使用统一的截尾均值计算
- 后续赛季根据样本量考虑优化
```

**设计理念**:
- **首届简化**: 不做复杂调整，保持系统简洁
- **数据优先**: 等待足够样本量后再优化
- **迭代优化**: 第二届开始考虑类型差异和人数权重

### ✅ Q4: 心理价位的填写时机问题 - 已确认

**问题描述**:

提交时填写心理价位 = 信息不对称，用户没有参考数据。

**具体场景**:
```
新手创作者:
- 第一次参赛
- 不了解平台平均打赏水平
- 看到建议"2-5金币"
- 随意填了3金币

结果:
- 游戏质量很高，实际收益8金币
- 吻合度33% → "被低估"📈
- 但用户真的很诚实地填了3金币！

这是运气好，不是"定价能力"
```

**选项**:

**A. 允许评分期开始前修改（推荐）**
```
规则:
- 提交时: 可以填写心理价位
- 评分期开始前: 可以随意修改
- 评分期开始后: 仅允许修改1次（在评分中期）
- 修改后重新计算吻合度（保留历史记录）
```
- 优点: 给用户学习和调整的机会
- 优点: 评分期开始后仍保留1次修改权
- 缺点: 可能导致用户频繁调整

**B. 提供智能参考数据**
```
UI改进:
- 显示"我的历史收益"参考
- 显示"同类型游戏平均收益"
- 给出智能建议: "根据您的游戏类型，建议价位X金币"
```
- 优点: 帮助用户做出更准确的判断
- 优点: 不强制限制
- 缺点: 新手仍无历史数据

**C. 评分期开始后统一填写**
```
规则:
- 提交时不填写心理价位
- 评分期开始后，看到初步收益后再填写
```
- 优点: 信息更充分
- 缺点: 可能失去"预期"的意义

**我的建议**: **选项A+B组合**
- 允许修改 + 提供智能参考
- 给用户最大的灵活性和信息支持

### ✅ Q5: 截尾均值的边界情况 - 已确认

**问题描述**:

打赏人数<10时使用算术均值，容易被刷单。

**具体场景**:
```
小众游戏: 只有5人打赏
- 正常: [1,2,2,3,5] → 均值2.6金币 ✅
- 刷单: [1,1,1,1,100] → 均值20.8金币 ❌

问题:
- 样本量小时，算术均值仍然容易被刷单
- 你的设计没有解决小样本的刷单问题
```

**选项**:

**A. 小样本强制过滤（推荐）**
```
规则:
- 打赏人数 < 10人:
  - 去掉最高和最低各1个（如果≥5人）
  - 或者直接使用中位数（如果<5人）
  - 强制标记"样本量小⚠️"

示例:
[1,1,1,1,100] → 去掉1和100 → 均值1金币 ✅
[1,2,100] → 使用中位数2金币 ✅
```
- 优点: 彻底解决小样本刷单问题
- 优点: 使用中位数抗极端值
- 缺点: 打赏人数很少时，数据可能不准确

**B. 提高截尾均值阈值**
```
规则:
- 打赏人数 < 20人: 使用中位数
- 打赏人数 ≥ 20人: 使用截尾均值
```
- 优点: 简单明确
- 缺点: 20人阈值可能太高

**C. 保持原设计**
```
打赏人数 < 10人: 使用算术均值
```
- 优点: 简单
- 缺点: 仍然存在刷单风险

**我的建议**: **选项A（小样本强制过滤）**
- 理由: 安全第一，宁可数据不准确也不要被刷单
- 标记"样本量小"提醒用户数据可信度低

---

**✅ 用户确认**: 问题不成立（门槛保证）

**最终规则**:

### ✅ Q6: 团队游戏的心理价位 - 已确认

**问题描述**:

团队游戏的心理价位由谁填写？如何处理分歧？

**具体场景**:
```
3人团队:
- 队长认为值10金币
- 成员A认为值5金币
- 成员B认为值15金币

问题:
1. 谁的意志是最终决定？
2. 吻合度算谁的？
3. 团队成员意见不统一怎么办？
```

**选项**:

**A. 仅队长填写，团队投票确认（推荐）**
```
规则:
1. 仅队长可以填写心理价位
2. 填写后，团队成员可以投票
   - 赞成: >50% → 确认
   - 反对: >50% → 队长重新填写
3. 吻合度记录到团队，不记录到个人
```
- 优点: 民主决策，尊重团队意见
- 优点: 防止队长独断
- 缺点: 增加决策复杂度

**B. 所有成员各自填写，取平均**
```
规则:
1. 所有成员各自填写心理价位
2. 系统取平均值作为团队心理价位
3. 吻合度计算使用平均值
4. 显示每个成员的预测偏差
```
- 优点: 充分表达每个人的意见
- 优点: 数据更丰富
- 缺点: 计算复杂，可能偏离真实预期

**C. 仅队长填写，团队确认**
```
规则:
1. 仅队长可以填写心理价位
2. 填写后，团队成员可以看到并确认
3. 不强制投票，但鼓励沟通
```
- 优点: 简单高效
- 缺点: 可能忽略成员意见

**我的建议**: **选项A（队长填写+投票）**
- 理由: 平衡效率和民主
- 团队决策应该体现集体意志

---

**✅ 用户确认**: 队长/副队长填写，组内决定

**最终规则**:
```
Q6简化处理:

规则:
- 权限: 仅队长或副队长可以填写心理价位
- 决策: 组内如何决定是他们自己的事
- 记录: 吻合度记录到团队（不记录到个人）

示例:
- 队长填写: 直接生效
- 副队长填写: 直接生效
- 组内沟通: 通过站内私聊自行协商
```

**设计理念**:
- **权限控制**: 队长/副队长拥有管理权限
- **内部自治**: 平台不干预团队内部决策
- **简化流程**: 避免复杂的投票机制

## 🟢 P2优先级 - 中优先级 (2个)

### ✅ Q7: 赛季结束后的数据时效性 - 已确认

**问题描述**:

赛季结束后，游戏打赏停止，心理价位还有意义吗？

```
赛季结束:
- 游戏A: 心理价10, 实际15, 吻合度50%

下一赛季:
- 游戏A被归档，打赏停止

**A. 按时间衰减权重（推荐）**
```
规则:
总吻合度 = (本赛季×3 + 上赛季×2 + 上上赛季×1) / 6
```
- 优点: 承认数据时效性递减
- 优点: 突出当前水平
- 缺点: 增加计算复杂度

规则:
- 历史数据归档，不参与计算

```
- 优点: 简单清晰
- 缺点: 丢失历史趋势信息

```
历史数据永久保留，不衰减
```
- 优点: 保留完整历史
- 缺点: 可能误导用户

**我的建议**: **选项A（时间衰减）**
- 理由: 承认数据时效性，更科学
- 权重可以根据实际情况调整
