### Q1: 游戏详情页的默认展开状态

**问题描述**:

游戏详情页包含大量信息（评分详情、打赏记录、收益统计等），默认应该如何展示？

**具体场景**:

**选项A: 全部展开**
- 默认展开所有模块
- 用户无需点击即可看到所有信息
- 缺点: 页面很长，信息过载

**选项B: 核心展开，次要收起**
- 评分统计卡片: 默认展开
- 收益统计卡片: 默认展开
- 评分详情: 默认收起（点击"查看详情"展开）
- 打赏记录: 默认收起（点击"查看记录"展开）
- 优点: 平衡信息量和页面长度

**选项C: 最小化展开**
- 仅展开游戏基本信息
- 其他所有卡片默认收起
- 优点: 页面简洁
- 缺点: 需要多次点击

**我的建议**: **选项B（核心展开，次要收起）**
- 理由: 平衡用户体验和信息获取效率
- 核心数据（评分、收益）是用户最关心的，应该直接展示
- 次要数据（详细记录）可以按需展开

### Q2: 评分页面的悬浮提示时机

**问题描述**:

评分标准悬浮提示应该在什么时候显示？

**具体场景**:

**选项A: 鼠标悬浮时显示**
- 鼠标移到滑块上才显示提示
- 优点: 界面简洁
- 缺点: 新手用户可能不知道有提示

**选项B: 鼠标按下时显示**
- 开始拖动滑块时显示提示
- 优点: 评分时即时看到标准
- 缺点: 评分前看不到标准

**选项C: 始终显示简略版，悬浮显示详细版**
- 滑块下方始终显示简要标准（如"8-9分: 高度创新"）
- 鼠标悬浮时显示详细标准
- 优点: 兼顾简洁和易用性
- 缺点: 占用更多空间

**选项D: 首次评分时自动展开，后续悬浮显示**
- 用户第一次评分时，自动展开所有评分标准
- 后续评分改为悬浮显示
- 优点: 新手友好，老手简洁
- 缺点: 实现复杂

**我的建议**: **选项C（始终显示简略版，悬浮显示详细版）**
- 理由: 平衡易用性和简洁性
- 简略版给用户即时反馈
- 详细版给需要深入理解的用户

### Q3: 评分页面的默认值设置

**问题描述**:

首次评分时，滑块的默认值应该是什么？

**具体场景**:

**选项A: 默认5.0分（中间值）**
- 所有维度默认5.0分
- 优点: 中性起点，鼓励用户主动调整
- 缺点: 用户可能直接提交默认值

**选项B: 默认0分（最低值）**
- 所有维度默认0分
- 优点: 强制用户主动评分
- 缺点: 可能给用户"默认是0分"的负面印象

**选项C: 随机默认值（4-6分之间）**
- 每个维度随机4-6分
- 优点: 避免默认值偏差
- 缺点: 可能混淆用户

**选项D: 无默认值，强制用户明确选择**
- 滑块显示为"未评分"状态
- 不提交无法提交
- 优点: 强制用户明确评分
- 缺点: 增加操作成本

**我的建议**: **选项A（默认5.0分）**
- 理由: 5.0分是中性、中性的起点
- 配合二次确认弹窗，提醒用户检查评分
- 用户心理上更容易接受从"一般"开始调整

### Q4: 打赏金额的自定义滑块范围

**问题描述**:

自定义打赏金额滑块应该支持什么范围？

**具体场景**:

**选项A: 1-10金币连续滑块**
- 滑块可以任意选择1-10之间的任何数值（支持小数）
- 优点: 灵活度高
- 缺点: 过于精细，用户难以选择

**选项B: 1-10金币整数滑块**
- 滑块只能选择整数（1, 2, 3, ..., 10）
- 优点: 简洁清晰
- 缺点: 灵活度较低

**选项C: 6-10金币整数滑块**（因为1-5已有单选按钮）
- 自定义滑块仅支持6-10
- 优点: 避免与1-5单选按钮重复
- 缺点: 不支持1-5的自定义

**选项D: 取消自定义滑块，仅提供1-10单选按钮**
- 点击按钮选择打赏金额
- 优点: 最简洁
- 缺点: 占用空间较多

**我的建议**: **选项B（1-10金币整数滑块）**
- 理由: 整数足够满足需求，小数意义不大
- 保持灵活性，同时避免过于复杂
- 可以考虑与1-5单选按钮联动（选中6-10时显示滑块）

### Q6: 评分页面的金币余额实时更新策略

**问题描述**:

评分页面的金币余额应该如何更新？

**具体场景**:

**选项A: 实时查询（每次调整滑块时查询）**
- 每次用户调整打赏金额，立即查询最新余额
- 优点: 数据最准确
- 缺点: 服务器压力大，可能有延迟

**选项B: 页面加载时查询，后续不更新**
- 页面加载时查询一次余额
- 后续调整打赏金额时不重新查询
- 优点: 减少服务器压力
- 缺点: 数据可能不准确（如果用户同时在其他地方消费）

**选项C: 页面加载时查询，每隔30秒更新一次**
- 页面加载时查询一次余额
- 后续每隔30秒自动更新一次
- 优点: 平衡准确性和性能
- 缺点: 仍然可能不准确

**选项D: 提交时验证，页面显示本地计算值**
- 页面显示的余额基于本地计算（初始余额 - 调整的打赏金额变化）
- 提交时才在服务器端验证余额
- 优点: 性能最好，用户体验流畅
- 缺点: 显示的余额可能不是最新的

**我的建议**: **选项D（提交时验证，页面显示本地计算值）**
- 理由: 性能和用户体验最佳
- 用户在评分页面不太可能同时在其他地方消费
- 提交时的服务器验证确保数据准确性
- 如果余额不足，提交时返回错误提示即可

### Q7: 游戏详情页的截图查看器交互方式

**问题描述**:

游戏截图应该如何查看和交互？

**具体场景**:

**选项A: 横向滚动画廊 + 点击查看大图**
- 默认显示横向滚动的缩略图
- 点击缩略图 → 模态框显示大图
- 优点: 经典模式，用户熟悉
- 缺点: 需要两次点击才能看到大图

**选项B: 点击缩略图直接切换主图**
- 默认显示一张大图 + 横向缩略图列表
- 点击缩略图 → 切换主图
- 优点: 一次点击即可查看
- 缺点: 占用空间较多

**选项C: 灯箱效果（Lightbox）**
- 点击任意截图 → 全屏灯箱效果
- 支持左右箭头切换
- 优点: 沉浸式体验
- 缺点: 可能与页面其他交互冲突

**选项D: 懒加载 + 虚拟滚动**
- 截图使用虚拟滚动，仅加载可见区域
- 优点: 性能最佳
- 缺点: 实现复杂，用户可能不习惯

**我的建议**: **选项A（横向滚动画廊 + 点击查看大图）**
- 理由: 经典模式，用户最熟悉
- 实现简单，兼容性好
- 模态框可以支持键盘导航和缩放

### Q8: 评分页面的草稿保存机制

**问题描述**:

草稿应该如何保存？保存在哪里？

**具体场景**:

**选项A: 仅保存在本地浏览器（localStorage）**
- 草稿保存在用户浏览器
- 优点: 简单，无需服务器支持
- 缺点: 换设备后无法恢复

**选项B: 同时保存在本地和服务器**
- 本地保存一份，服务器保存一份
- 优点: 换设备后可恢复
- 缺点: 实现复杂，服务器压力大

**选项C: 仅保存在服务器**
- 草稿保存在用户账户中
- 优点: 换设备后可恢复
- 缺点: 需要登录，服务器压力大

**选项D: 不保存草稿**
- 用户离开页面后草稿丢失
- 优点: 最简单
- 缺点: 用户可能意外丢失数据

**我的建议**: **选项A（仅保存在本地浏览器）**
- 理由: 草稿是临时性的，不需要跨设备同步
- 减少服务器压力
- 实现简单，用户体验足够

### Q10: 评分页面的部分评分提示

**问题描述**:

用户仅对部分维度评分时，应该如何提示？

**具体场景**:

**选项A: 仅在确认弹窗中提示**
- 用户点击"提交评分"时，在确认弹窗中列出未评分的维度
- 优点: 不干扰评分过程
- 缺点: 用户可能直到最后才发现

**选项B: 实时在页面上标记未评分维度**
- 未评分的维度显示"未评分"标签
- 页面顶部显示"您有3个维度未评分"提示
- 优点: 提醒及时
- 缺点: 可能给用户压力

**选项C: 进度条显示评分进度**
- 页面顶部显示进度条（如"评分进度: 3/6"）
- 优点: 鼓励用户完成评分
- 缺点: 可能让用户觉得是任务

**选项D: 不提示，允许部分评分**
- 完全不提示，允许用户自由选择评分维度
- 优点: 最大自由度
- 缺点: 用户可能不知道建议全面评分

**我的建议**: **选项A+B组合（实时标记 + 确认弹窗提示）**
- 理由: 兼顾及时性和明确性
- 实时标记让用户知道哪些未评分
- 确认弹窗给出最后提示
- 最终允许部分评分提交（不强制）

### Q12: 评分页面的滑块颜色反馈

**问题描述**:

滑块应该如何根据分数变化颜色？

**具体场景**:

**选项A: 根据分数档位变色**
- 0-3分: 红色
- 4-5分: 橙色
- 6-7分: 黄色

- 10分: 金色
- 优点: 直观反映分数等级
- 缺点: 可能让用户产生"红色=不好"的压力

**选项B: 统一品牌色**

- 优点: 简洁，避免颜色歧视
- 缺点: 缺乏视觉反馈

**选项C: 渐变色彩**
- 滑块使用渐变色（从红到绿）
- 颜色反映分数高低
- 优点: 美观
- 缺点: 实现复杂

**选项D: 仅滑块头（滑块按钮）变色**
- 滑块轨道不变色
- 仅滑块按钮根据分数变色
- 优点: 简洁且有反馈
- 缺点: 可能不够明显

**我的建议**: **选项B（统一品牌色）**
- 理由: 避免颜色带来的心理暗示
- 保持界面简洁统一
- 评分应该基于用户自己的判断，而不是颜色引导

### Q14: 评分页面的键盘支持

**问题描述**:

评分页面应该支持哪些键盘操作？

**具体场景**:

**选项A: 基础键盘支持**
- Tab: 切换焦点
- Enter: 提交
- Esc: 取消
- 左右箭头: 调整滑块
- 优点: 基础可访问性
- 缺点: 功能有限

**选项B: 扩展键盘支持**
- 基础支持 + 数字键直接输入分数
- 优点: 高级用户更高效
- 缺点: 可能冲突（数字键可能用于其他功能）

**选项C: 完整键盘快捷键**
- 自定义快捷键（如S=保存，C=取消等）
- 优点: 高效
- 缺点: 学习成本高

**选项D: 不支持键盘**
- 仅支持鼠标操作

### Q32: 游戏详情页比赛结果展示区域设计 ⭐⭐⭐ (新增)

**问题描述**:

参考 Ludum Dare 的游戏详情页结果展示区域,如何设计惊蛰计划的比赛结果展示?

**Ludum Dare 参考图**:

用户提供的 Ludum Dare 游戏详情页结果展示区域显示了不同评分类别的数据,采用 Tab 切换的方式组织信息。

**设计方案**: **三个 Tab 切换展示**

参考 Ludum Dare 的设计,将比赛结果分为三个可切换的子表:
1. **玩家评分** - 普通玩家评分统计
2. **评委评分** - 评审团评分统计
3. **打赏结果** - 收益和心理价位信息

---

#### Tab 1: 玩家评分

**显示内容**:
- 6个维度的截尾平均分
- 横向排名(在该维度排第几)
- 评分人次
- 进度条可视化评分百分比
