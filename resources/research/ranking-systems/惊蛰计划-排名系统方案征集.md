## 📋 征集背景

### 惊蛰计划是什么？

惊蛰计划是一个**48小时游戏开发竞赛平台**，核心特色是：

**🎮 革命性排名系统**：
- **12个独立榜单**（6个评分维度 × 2个评分群体）
- **不计算总分**，避免权重争议
- **单维度排名**，保持各维度纯粹性

**👥 三种角色**：
- **参赛者**：提交游戏，互相评分
- **评审团**：专家评审，评分权重更高
- **路人玩家**：观看、打赏，不评分

**⚖️ 评分规则**：
- 每个维度0-10分
- 支持0.5分档位
- 评审团评分权重 ×2

## 🎯 征集目标

我们需要一个**排名计算方法**，能够：

### 核心需求

1. **公平性** ⭐⭐⭐
   - 真实反映游戏质量
   - 不容易被刷分操纵
   - 认真评分的人权重更高

2. **可解释性** ⭐⭐⭐
   - 参赛者能理解为什么自己排这个名次
   - 计算逻辑透明，不黑箱
   - 便于沟通规则

3. **可实现性** ⭐⭐
   - 第一届就能落地
   - 技术复杂度可控
   - 计算性能可接受（<1秒）

4. **数据驱动** ⭐
   - 评分越多，结果越可信
   - 鼓励互相评分
   - 避免冷启动问题

## ❓ 当前方案的问题

### 简单算术平均的问题

**当前方案**：
```
维度得分 = 该维度所有评分的算术平均值
```

**存在的问题**：

#### 问题1：极端值影响大
- 1个恶意1分可以拉低整体得分
- 无法自动识别刷分行为

**场景**：
- 游戏A真实水平8分
- 被恶意刷1个1分
- 算术平均：`(8×10 + 1×1) / 11 = 7.36` ❌ 失真

#### 问题2：评分贡献度不同
- 评分100次的用户 vs 评分1次的用户
- 两者权重相同，不公平

**场景**：
- 用户A：认真评分100次
- 用户B：随意评分1次
- 他们的评分权重相同 ❌

#### 问题3：刷成本低
- 只要注册账号就能刷分
- 不需要长期参与社区

## 💡 我们的初步想法（供参考）

### 方向1：评分者权重法

**核心思想**：**评分越多的人，权重越高**

**权重公式**（示例）：
```
用户权重 = 1 + log(评分次数)
```

**示例**：
- 评分1次：权重 = 1.0
- 评分5次：权重 ≈ 1.7
- 评分10次：权重 = 2.0
- 评分50次：权重 ≈ 2.7

**计算公式**：
```
维度得分 = Σ(评分 × 用户权重) / Σ(用户权重)
```

**优点**：
- ✅ 公平：认真评分的人权重更高
- ✅ 可解释：容易理解"评分越多权重越高"
- ✅ 抗刷：刷分账号权重低，影响小
- ✅ 数据驱动：鼓励多评分

**缺点**：
- ⚠️ 计算稍复杂（但可接受）
- ⚠️ 需要实时更新用户权重

---

### 方向2：截尾均值法

**核心思想**：**去掉最高和最低各10%的评分**

**计算步骤**：
1. 收集所有评分
2. 去掉最高的10%
3. 去掉最低的10%
4. 剩余的评分算术平均

**示例**：
- 100个评分：去掉10个最高 + 10个最低
- 剩余80个评分算平均

**优点**：
- ✅ 简单：逻辑清晰
- ✅ 抗刷：自动过滤极端值
- ✅ 可解释：容易理解

**缺点**：
- ⚠️ 评分少时效果差（如只有5个评分）
- ⚠️ 不区分评分者贡献度

---

### 方向3：贝叶斯平滑法

**核心思想**：**新游戏趋向全场平均，评分多后逐渐收敛**

**公式**：
```
S' = (n × S + k × μ) / (n + k)
```

其中：
- S = 当前游戏算术平均
- n = 实际评分数
- μ = 系统全场平均
- k = 平滑系数（如5）

**优点**：
- ✅ 科学：成熟的统计方法
- ✅ 冷启动友好：新游戏不会被埋没
- ✅ 抗刷：刷分影响会被"拉回"

**缺点**：
- ⚠️ 复杂：难向参赛者解释
- ⚠️ 可能不符合"革命性排名"的初衷

## 🎨 我们期待的方案

### 理想方案的特征

1. **结合方向1和方向2**
   - 使用**评分者权重**（鼓励认真评分）
   - 引入**截尾均值**（抗刷分）
   - 保持**可解释性**（参赛者能理解）

2. **参数可调**
   - 第一届用简单参数
   - 后续根据数据优化

3. **计算高效**
   - 单个游戏排名计算 < 100ms
   - 全平台排名重算 < 10秒
