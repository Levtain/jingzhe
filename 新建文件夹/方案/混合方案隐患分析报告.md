### 隐患1: 权重计算导致分数膨胀 ⚠️⚠️⚠️

#### 问题描述

**当前算法步骤4的还原逻辑有严重缺陷**:

```python
# 当前文档中的错误做法
final_score = sum(trimmed_scores) / len(trimmed_scores)  # 加权平均
avg_weight = sum(weights) / len(weights)
final_score_normalized = final_score / avg_weight  # 还原
```

**问题在哪里？**

假设场景：
- 用户A（权重4.0，认真评分20次）给出10分
- 用户B（权重1.7，只评1分）给出1分

**加权后**:
- A: 10 × 4.0 = 40
- B: 1 × 1.7 = 1.7

**如果这两个分数都被截尾保留**:
- 加权平均 = (40 + 1.7) / 2 = 20.85
- 平均权重 = (4.0 + 1.7) / 2 = 2.85
- **还原后 = 20.85 / 2.85 = 7.32** ✅ 看似正确

**但如果只有A的高分被截尾保留**:
- 截尾去掉了B的1分
- 加权平均 = 40 / 1 = 40
- 平均权重 = 4.0 / 1 = 4.0
- **还原后 = 40 / 4.0 = 10.0** ✅ 正确

**但如果只有B的低分被截尾保留**:
- 截尾去掉了A的40分
- 加权平均 = 1.7 / 1 = 1.7
- 平均权重 = 1.7 / 1 = 1.7
- **还原后 = 1.7 / 1.7 = 1.0** ✅ 正确

**看起来没问题？继续看复杂场景...**

### 隐患2: 权重还原的数学问题 ⚠️⚠️

#### 问题描述

**当前算法**:
```python
加权评分 = 原始评分 × 权重
加权平均 = sum(加权评分) / n
平均权重 = sum(权重) / n
最终得分 = 加权平均 / 平均权重
```

**数学等价性**:

设原始评分为 x₁, x₂, ..., xₙ
权重为 w₁, w₂, ..., wₙ

加权平均 = (Σ wᵢxᵢ) / n
平均权重 = (Σ wᵢ) / n
最终得分 = (Σ wᵢxᵢ) / n / [(Σ wᵢ) / n] = (Σ wᵢxᵢ) / (Σ wᵢ)

**这其实就是加权算术平均！** ✅

**所以还原公式数学上是正确的**

**但问题在哪里？**

问题在于:**截尾破坏了权重的代表性**

---

### 隐患4: 小样本截尾问题 ⚠️

#### 问题描述

**当前规则**:
- n < 10: 不截尾
- n ≥ 10: 截尾10%

**问题**: **n=10-20时，截尾后只剩9-18个评分**

---

#### 场景测试

**游戏刚好达到门槛**:
- 11个评分
- 截尾10%: k = int(11 × 0.1) = 1
- 去掉1个最高，1个最低
- 剩余: 9个评分

**问题**: 样本量从11降到9，损失18%的信息

**更极端**:
- 10个评分
- 截尾10%: k = max(1, int(10 × 0.1)) = 1
- 去掉1个最高，1个最低
- 剩余: 8个评分
- 损失20%的信息

**如果10个评分中有2个是极端值**:
- 真实评分: 8个8分
- 刷分评分: 2个1分
- 截尾去掉2个1分 ✅ 有效
- 剩余8个8分 ✅ 结果正确

**但如果10个评分中有3个是极端值**:
- 真实评分: 7个8分
- 刷分评分: 3个1分
- 截尾只能去掉1个1分 ❌
- 剩余: 2个1分 + 7个8分
- **最终得分被拉低** ❌

**结论**: 小样本时截尾效果有限

### 隐患5: 权重激励不足 ⚠️

#### 问题描述

**当前权重公式**: W = 1 + ln(1 + n)

**权重增长**:
- 1次: 1.69
- 2次: 2.10 (+0.41)
- 5次: 2.79 (+0.69)
- 10次: 3.39 (+0.60)
- 20次: 4.04 (+0.65)
- 50次: 4.93 (+0.89)

**问题**: **权重增长递减，边际效应明显**

---

#### 激励效果分析

**用户A**: 从1次评到10次
- 权重: 1.69 → 3.39
- 增长: 2×
- 付出: 多评9次
- **每次评分的权重增长**: (3.39-1.69)/9 = 0.19

**用户B**: 从10次评到50次
- 权重: 3.39 → 4.93
- 增长: 1.45×
- 付出: 多评40次
- **每次评分的权重增长**: (4.93-3.39)/40 = 0.0385

**问题**: 用户B的努力是用户A的4.4倍（40次 vs 9次），但每次评分的权重增长只有1/5（0.0385 vs 0.19）

**结论**: **对高频评分用户的激励不足**

---

#### 更严重的问题

**评分1次 vs 评分100次**:
- 1次: 权重 = 1.69
- 100次: 权重 = 1 + ln(101) ≈ 5.62
- 差距: 3.32×

**但评分100次需要**:
- 如果100个游戏，每个游戏评6个维度 = 600次评分
- 时间成本: 假设每个评分30秒 = 5小时
- **权重增长3.32倍，但付出100倍努力**

**问题**: **投入产出比极低**

## 💡 轻微隐患（可以接受）

### 隐患9: 计算性能 ⚠️

#### 当前评估

**文档声称**: <1ms计算单维度

**问题**: **没有考虑权重更新的性能**

---

#### 权重更新流程

**每次有新评分时**:
1. 更新用户的评分次数: n → n+1
2. 重新计算权重: W = 1 + ln(1 + (n+1))
3. **重新计算该用户之前的所有评分的权重**
4. **重新计算所有受影响的游戏的排名**

**性能问题**:
- 用户A评了50次，权重从4.93更新到4.94
- 用户A之前评过的49个游戏的权重都要更新
- 这49个游戏的排名都要重新计算

**结论**: 增量更新并不简单

**缓解**: 定时全量重算（每小时一次），而不是增量更新

## 💡 解决方案建议

### 针对隐患1: 刷分账号超过20%

**方案A: 提高截尾比例**
- 当前: 10%
- 建议: 20-25%
- 代价: 正常评分也被截掉更多

**方案B: 多轮截尾**
- 第1轮: 截尾10%
- 第2轮: 检查剩余评分的标准差
- 如果标准差过大，再截尾10%
- 代价: 复杂度增加

**方案C: 基于权重的截尾**
- 不截尾"加权后"的分数
- 而是: 去掉权重最低的10%评分 + 权重最高的10%评分
- 代价: 更难解释

**方案D: 人工监控 + 后台标记**
- 检测异常评分模式
- 人工审核
- 取消作弊者资格

### 针对隐患6: 冷启动问题

**方案**: 引入简化的贝叶斯平滑

```python
# 简化版贝叶斯平滑
if n < 20:  # 评分较少时
    k = 20  # 平滑系数
    mu = 7.5  # 全平台平均（可预设）
    score = (n * calculated_score + k * mu) / (n + k)
else:
    score = calculated_score
```

**代价**: 轻微增加复杂度

**收益**: 新游戏更公平

**建议**: 第二届引入

### 针对隐患7: 并列处理

**修改规则**:

**并列处理优先级**:
1. 评审团得分高的优先
2. **加权评分人次高的优先** = Σ 权重
3. 创新性得分高的优先
4. 允许并列

**示例**:
- 游戏A: 10个评分，平均权重2.0，加权人次 = 20
- 游戏B: 10个评分，平均权重4.0，加权人次 = 40
- 游戏B优先 ✅

### 针对隐患8: 有效评分定义

**明确定义**:

**有效评分**:
- 用户对**不同游戏**的评分
- 每个游戏**每个维度**只计算**最新一次**评分
- 修改评分后，旧评分失效

**统计方式**:
```sql
SELECT COUNT(DISTINCT CONCAT(game_id, dimension))
FROM scores
WHERE user_id = ?
```

**防止刷权重**:
- 一个用户对同一个游戏同一个维度多次评分
- 只计算最新一次
- 防止通过重复评分刷权重
