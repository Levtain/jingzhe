## 一、核心设计原则

### 1.1 设计目标
> **激励参与、促进互动、形成循环**

### 1.2 核心理念
- **有成本的互动更认真**: 评分必须打赏，形成成本约束
- **净成本为0**: 奖励与打赏平衡，不阻碍参与
- **经济循环**: 金币从评分者流向游戏开发者，激励创作
- **适度激励**: 不过度强调金币，保持比赛的好玩性

## 三、金币获取方式

### 3.1 评分奖励

#### 3.1.1 有效评分定义

**定义**: 对某个游戏的第一次评分提交（无论评了几个维度）

**说明**:
- ✅ 第一次提交即可，无论评了几个维度
- ✅ 允许对某些维度给0分
- ❌ 修改和重复不计入
- ❌ 只计算第一次提交

#### 3.1.2 奖励规则

**奖励金额**: 每次有效评分获得1金币

**触发时机**: 评分提交成功后立即到账

**限制**:
- 每个游戏只能获得1次奖励
- 修改评分不再获得奖励
- 评分被删除后回收奖励

**参考决策**: [排名系统决策记录.md - 决策14](./排名系统决策记录.md)

**条件**: 每日投出总主题量1/3的赞

#### 3.2.2 奖励规则

**奖励金额**: 5金币

**触发时机**: 每日投票任务完成后立即到账

### 3.4 赛后结算 ✅ 已确认

#### 3.4.1 结算规则

**规则**: 赛后获得已收到赏金等额金币

**适用范围**: **仅适用于参赛者**（个人参赛者、团队参赛者）

**结算时间**: 评分阶段结束后，结果公布前

**结算方式**: 一次性发放所有冻结金币

#### 3.4.2 结算计算公式

**个人参赛者**:
```
个人收益 = 游戏赏金 + 个人评论赏金

示例:
- 游戏获100赏金 + 个人评论获20赏金 = 120金币
```

**团队参赛者**:
```
个人收益 = 团队游戏赏金(全额) + 个人评论赏金

示例:
- 团队获100游戏赏金 + 个人A获20评论赏金 = A得120金币
- 团队获100游戏赏金 + 个人B获30评论赏金 = B得130金币
- 团队获100游戏赏金 + 个人C获10评论赏金 = C得110金币
```

**关键特征**:
1. **游戏赏金全额给每位团队成员**: 团队获100赏金 → 3人团队每人得100（不是100/3=33）
2. **评论赏金个人独得**: 不会被团队平分
3. **激励效果**: 鼓励团队参赛，团队协作获得更多游戏赏金

#### 3.4.4 心理价位与收益展示 ✅ 已确认

**定义**: 创作者在游戏提交时设定的心理预期价格，用于与实际收益对比。

**填写规则**:
- **填写时机**: 游戏提交流程步骤6.6（未完成作品选项之后，提交确认之前）
- **填写权限**:
  - 个人参赛: 仅作者本人可以填写
  - 团队参赛: 仅队长或副队长可以填写
- **数值范围**: 1-20金币
- **修改规则**:
  - 评分阶段开始前: 可以随意更改
  - 评分阶段开始后: 非特殊情况不可更改（参考"更改游戏链接"规则）

**展示值计算 - 截尾均值**:
```python
def trimmed_mean(tips):
    """
    计算截尾均值
    tips: 打赏金额列表
    """
    n = len(tips)

    # 去掉最高和最低各10%
    k = int(n * 0.1)
    sorted_tips = sorted(tips)
    trimmed = sorted_tips[k:n-k]

    return sum(trimmed) / len(trimmed)
```

**边界规则**:
- 打赏人数 ≥ 20人: 去掉最高/最低各10%
- 打赏人数 10-19人: 去掉最高/最低各1个
- 打赏人数 < 10人: 不适用（15人次评分门槛保证样本充足）

**吻合度计算公式**:
```
吻合度 = max(0, 100 - 10 × |展示值 - 心理价|) %
```

**示例**:
- 心理价10金币，展示值10金币 → 吻合度 = 100%
- 心理价10金币，展示值9金币 → 吻合度 = 90%
- 心理价10金币，展示值5金币 → 吻合度 = 50%
- 心理价10金币，展示值0金币 → 吻合度 = 0%
- 心理价10金币，展示值20金币 → 吻合度 = 0% (max函数保护)

**溢价/折价标签**:
```
if 展示值 > 心理价 + 1:
    标签 = "被低估 📈"
    if 展示值 - 心理价 >= 3:
        标签 += " ⚠️" (红色叹号)
elif 展示值 < 心理价 - 1:
    标签 = "被高估 📉"
    if 心理价 - 展示值 >= 3:
        标签 += " ⚠️" (红色叹号)
else:
    标签 = "符合预期 ✅"
```

**展示位置**:
1. **游戏详情页**: 收益统计卡片
2. **个人作品分析面板**: 自我认知分析
3. **赛季总结报告**: 个人定价能力成长

**数据时效性**:
- 赛季结束后: 不可以评分，不可以打赏
- 数据作用: 仅供展示，无时效性衰减
- 展示方式: 每个赛季的数据独立展示

**设计原则**:
- **首届简化**: 不做复杂调整，保持系统简洁
- **用户自主**: 尊重用户的判断和决定
- **话题制造**: "被低估"/"被高估"标签自然引发讨论

**详细设计文档**: [economic-system-price-mechanism-design_2025-01-09.md](../../development/analysis/economic-system-price-mechanism-design_2025-01-09.md)
**问题讨论记录**: [economic-system-price-mechanism-questions_2025-01-09.md](../../development/analysis/economic-system-price-mechanism-questions_2025-01-09.md)

#### 3.4.5 金币流向

**赛前**:
- 评分者金币 → 打赏给游戏开发者
- 游戏开发者金币 → 购买推荐位等消费

**赛中**:
- 评分者金币 → 打赏给游戏开发者
- 游戏开发者收到打赏（暂存）

**赛后**:
- 游戏开发者收到的打赏 → 等额金币结算到账户
- 目的: 确保开发者真正获得收益，激励创作

## 四、金币消费方式

### 4.1 评分打赏

#### 4.1.1 打赏规则

**强制打赏**: 评分后必须打赏至少1金币

**打赏档位**: 1-10金币（共10档）

**默认档位**: 1金币（减少操作成本）

**限制**:
- 不打赏无法提交评分
- 可以调整打赏金额
- 打赏金额从账户余额扣除

#### 4.1.2 打赏流程

```
1. 用户完成评分
   ↓
2. 系统显示打赏界面（默认1金币）
   ↓
3. 用户调整打赏金额（1-10金币）
   ↓
4. 用户确认提交评分+打赏
   ↓
5. 系统扣除评分者金币
   ↓
6. 系统给游戏开发者增加打赏记录
   ↓
7. 评分提交成功
```

#### 4.1.3 打赏记录

**记录内容**:
- 打赏者ID
- 收到打赏的游戏ID
- 打赏金额
- 打赏时间
- 关联的评分ID

**参考决策**: [排名系统决策记录.md - 决策13](./排名系统决策记录.md)

## 五、经济模型

### 5.1 金币流向图

```
┌─────────────────────────────────────────────┐
│                  金币循环                     │
└─────────────────────────────────────────────┘

                评分者
                  │
                  │ +1金币 (评分奖励)
                  │
                  ↓
              提交评分
                  │
                  │ -1金币 (评分打赏)
                  │
                  ↓
              游戏开发者
                  │
                  │ 收到打赏 (暂存)
                  │
                  ↓
              赛后结算
                  │
                  │ +等额金币
                  │
                  ↓
              开发者账户
                  │
                  │ 购买推荐位等消费
                  │
                  ↓
              流向其他用户
                  │
                  │ ────────────┐
                  │             │
                  └─────────────┘
                   形成经济循环
```

### 5.2 净成本计算

#### 5.2.1 评分者净成本

**计算公式**:
```
净成本 = 评分打赏 - 评分奖励
      = 1金币 - 1金币
      = 0金币
```

**说明**:
- 评分的净成本为0金币
- 目的: 激励评分但不阻碍参与
- 实际: 评分者需要先有金币才能评分

#### 5.2.2 游戏开发者收益

**计算公式**:
```
个人参赛者:
收益 = 收到的游戏赏金 + 收到的评论赏金

团队参赛者:
个人收益 = 团队游戏赏金(全额) + 个人评论赏金
```

**说明**:
- 开发者收到的打赏暂存
- 赛后结算为等额金币
- 团队游戏赏金全额给每位成员（不按人头平分）
- 评论赏金个人独得（不被团队平分）
- 目的: 激励创作，鼓励团队协作，形成经济循环

**示例对比**:

| 参赛类型 | 游戏赏金 | 评论赏金 | 个人收益 |
|---------|---------|---------|---------|
| 个人参赛 | 100 | 20 | 120 |
| 团队参赛(成员A) | 团队100 | 20 | 120 |
| 团队参赛(成员B) | 团队100 | 30 | 130 |
| 团队参赛(成员C) | 团队100 | 10 | 110 |

**设计理念**:
- 团队协作更容易创作出优质游戏
- 全额分配团队赏金是对团队协作的额外激励
- 评论赏金个人独得，鼓励每个成员都参与评分

## 十、与相关系统的接口

### 10.1 与评分系统的接口

**评分系统 → 经济系统**:
- 评分提交成功后，通知经济系统发放奖励
- 评分打赏扣除金币

**经济系统 → 评分系统**:
- 返回金币余额，判断是否可以评分

### 10.2 与排名系统的接口

**排名系统 → 经济系统**:
- 排名公布后，触发排名奖励（如果有的话）

**经济系统 → 排名系统**:
- 无直接依赖

### 10.3 与成就系统的接口

**经济系统 → 成就系统**:
- 交易记录触发成就解锁（打赏10次、打赏总额100金币等）

**成就系统 → 经济系统**:
- 成就解锁发放金币奖励

## 十一、技术实现要点

### 11.1 事务处理

**评分+打赏提交**:
```
BEGIN TRANSACTION
  1. 验证金币余额是否足够
  2. 扣除评分者金币
  3. 创建打赏记录（冻结状态）
  4. 创建评分记录
  5. 发放评分奖励
  6. 记录交易日志
COMMIT
```

**异常处理**:
- 任何步骤失败，回滚整个事务
- 确保金币不会丢失或重复发放

### 11.2 并发控制

**金币余额更新**:
- 使用乐观锁或悲观锁
- 防止并发更新导致余额错误

**打赏记录**:
- 幂等性设计，防止重复提交
- 唯一索引约束

### 11.3 性能优化

**缓存策略**:
- 缓存用户金币余额
- 缓存游戏打赏总额

**异步处理**:
- 交易记录异步写入
- 打赏通知异步发送
