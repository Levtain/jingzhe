## 示例3：遇到技术问题

**Claude**：
【进度汇报】
✅ 已完成：评分页面UI（80%）
🔄 进行中：评分数据保存（遇到问题）
⏳ 待开始：评分统计

【遇到的问题】
问题：评分人数统计不准确

【问题说明】
现在的逻辑是：
- 参赛者A给游戏B评分 → 评分人数+1
- 参赛者A修改给游戏B的评分 → 评分人数又+1（不对！）

这就像：
- 第一次投票，计票器记录了
- 修改投票，计票器又记了一次（重复计数了）

【可能的方案】
方案1：检查是否已经评过分
- 优点：准确
- 缺点：需要查询数据库，速度稍慢
- 时间：需要30分钟修改

方案2：用唯一索引
- 优点：数据库层面防止重复
- 缺点：需要调整表结构
- 时间：需要1小时

方案3：前端判断
- 优点：速度快
- 缺点：不能完全防止（可能绕过前端）
- 时间：需要10分钟

【我的建议】
先用方案1（检查是否已评分），最稳妥。
如果以后发现性能问题，再优化到方案2。

你觉得呢？

**产品经理**：
听你的，用方案1。

**Claude**：
【收到确认】
好的，我现在修改评分逻辑。

【修改内容】
1. 保存评分前，先查询是否已经评过
2. 如果评过，更新原评分
3. 如果没评过，新增评分并计数+1

30分钟后修改完成，再向你汇报。

## 总结

### 进度汇报的关键点

1. **清晰的状态标识**
   - ✅ 已完成
   - 🔄 进行中（标注百分比）
   - ⏳ 待开始

2. **具体的成果描述**
   - 做了什么（用非技术语言）
   - 完成了哪些功能

3. **诚实的问题说明**
   - 遇到什么问题
   - 为什么会出现
   - 如何解决的

4. **明确的时间说明**
   - 原计划时间
   - 实际进度
   - 调整后的时间

5. **主动的配合请求**
   - 需要用户什么
   - 为什么需要
   - 何时需要

### 常用话术

- "已完成..."
- "进行中..."
- "遇到的问题..."
- "下一步..."
- "需要你配合..."
- "预计...完成"
