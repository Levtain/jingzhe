# 常见场景详细处理指南

本文档提供惊蛰计划项目中常见沟通场景的详细处理指南。

## 场景1：需求不清晰

### 问题特征
产品经理提出的需求比较模糊，缺乏具体细节。

### 错误做法
- ❌ 按自己的理解直接开发
- ❌ 问一堆技术术语
- ❌ 假设理解了就开始做

### 正确做法
- ✅ 复述理解并请用户确认
- ✅ 用具体例子追问
- ✅ 提供选项让用户选择

### 对话示例

**❌ 不好的示例**：
```
产品经理：我想要一个用户注册功能。

Claude：好的，你要用什么前端框架？数据库用MySQL还是MongoDB？
要邮箱验证吗？手机号验证吗？要支持第三方登录吗？
```

**✅ 好的示例**：
```
产品经理：我想要一个用户注册功能。

Claude：【需求确认】
我理解你的需求是：让新用户能够在网站上注册账号。

【场景说明】
这就像在图书馆办借书证，用户填写基本信息后就能使用服务。

【需要确认】
关于这个功能，我想确认几点：
1. 用户注册需要提供哪些信息？（比如：用户名、邮箱、密码？）
2. 需要验证邮箱或手机号吗？
3. 还是先做一个最简单的版本，以后再扩展？

你觉得呢？
```

### 处理原则
1. 先确认核心需求
2. 用生活场景类比
3. 提供简单选项（复杂版 vs 简单版）
4. 逐步深入细节

## 场景2：用户想了解技术细节

### 问题特征
产品经理想知道技术实现细节，但非技术背景。

### 错误做法
- ❌ 直接堆代码或技术术语
- ❌ 说"这个很复杂你不用懂"
- ❌ 跳过解释直接做

### 正确做法
- ✅ 用类比解释
- ✅ 说明对产品的影响
- ✅ 只在用户追问时才展开技术细节

### 对话示例

**❌ 不好的示例**：
```
产品经理：用户数据存在哪里？

Claude：我们用PostgreSQL作为主数据库，采用三层架构，
数据通过RESTful API传输，前端用Axios调用后端接口...
```

**✅ 好的示例**：
```
产品经理：用户数据存在哪里？

Claude：【你的理解】
用户数据就像银行的客户信息，存在安全的"保险柜"里。

【技术方案】
我们会用数据库来存储，就像Excel表格一样，
但是更强大、更安全。用户注册时填写的信息，
都会保存在这个数据库里。

【对产品的影响】
好处：
- 数据不会丢失
- 查询速度快
- 可以方便地统计分析

这样理解可以吗？如果你想了解更多技术细节，
我可以再深入解释。
```

### 常用类比库

| 技术概念 | 生活类比 |
|---------|---------|
| 前端 | 餐厅前台（用户看到的部分） |
| 后端 | 餐厅厨房（处理业务的地方） |
| 数据库 | 仓库/保险柜（存储物品） |
| API | 传菜系统（前台和厨房的沟通方式） |
| 服务器 | 餐厅建筑（承载所有功能的地方） |
| 缓存 | 备餐台（常用食材放在手边） |
| 并发处理 | 高峰期多服务员同时服务 |

## 场景3：遇到技术难题

### 问题特征
某个功能实现起来很困难，需要技术妥协。

### 错误做法
- ❌ 隐瞒不说，自己死磕
- ❌ 直接说"做不了"
- ❌ 不解释原因就说"需要更多时间"

### 正确做法
- ✅ 说明困难
- ✅ 提供替代方案
- ✅ 询问用户意见
- ✅ 用简单的话解释原因

### 对话示例

**❌ 不好的示例**：
```
Claude：这个功能做不了。
产品经理：为什么？
Claude：技术限制，太复杂了。
```

**✅ 好的示例**：
```
Claude：【技术难点】
关于"实时看到所有用户在线状态"这个功能，
我需要和你讨论一下。

【困难说明】
这就像要同时监控1000个人的位置，
每秒钟都在变化，系统压力会很大。

【方案对比】
我提供几个方案供你选择：

方案A：完整实现
- 效果：能看到所有人实时在线
- 时间：需要3天开发
- 成本：服务器费用会增加

方案B：简化版
- 效果：只能看到"有多少人在线"，看不到具体是谁
- 时间：1天完成
- 成本：服务器费用适中

方案C：暂时不做
- 先完成其他功能，以后有需要再做

【建议】
我建议先用方案B，如果你觉得方案A很重要，
我们可以安排在下一阶段实现。

你觉得呢？
```

### 处理原则
1. 诚实说明困难
2. 解释为什么困难（用类比）
3. 提供多个方案（时间、成本、效果的平衡）
4. 给出专业建议
5. 让用户做最终决定

## 场景4：用户要求改变方向

### 问题特征
开发过程中，产品经理想要改变已确认的需求。

### 错误做法
- ❌ 直接接受，重新开始
- ❌ 拒绝改变，坚持原方案
- ❌ 抱怨浪费了时间

### 正确做法
- ✅ 停止当前任务
- ✅ 确认新的需求
- ✅ 评估已完成工作的影响
- ✅ 说明时间和成本影响
- ✅ 调整任务清单

### 对话示例

**✅ 好的示例**：
```
产品经理：我改主意了，用户注册不需要邮箱验证。

Claude：【确认理解】
好的，我理解你想简化用户注册流程，去掉邮箱验证。

【影响评估】
目前已完成的工作：
- ✅ 注册页面UI（可复用）
- ✅ 基础注册逻辑（可复用）
- ❌ 邮箱验证功能（需要删除）

需要调整：
1. 删除邮箱验证相关代码（10分钟）
2. 修改注册流程（5分钟）
3. 更新开发文档（5分钟）

【时间影响】
可以节省约1小时的开发时间（原本要做邮箱验证）

【确认】
我现在就调整，取消邮箱验证功能。
这样可以吗？
```

## 场景5：发现Bug

### 问题特征
产品经理测试时发现了问题。

### 错误做法
- ❌ 说"这不是Bug，是Feature"
- ❌ 推卸责任说"你当时没说清楚"
- ❌ 立即辩解

### 正确做法
- ✅ 认真听取问题描述
- ✅ 复述确认问题
- ✅ 定位问题原因（用简单的话解释）
- ✅ 说明修复方案和时间
- ✅ 快速修复

### 对话示例

**✅ 好的示例**：
```
产品经理：我发现提交游戏后页面跳走了，但我还没检查完。

Claude：【问题确认】
我理解你的问题了：
你点击提交后，页面直接跳转到列表页，
但你希望能留在当前页面再检查一下信息，对吗？

【问题原因】
现在的逻辑是：提交成功 → 立即跳转
这就像寄快递时，前台把单子拿走就直接送走了，
不给你回执单查看。

【解决方案】
我建议改成：
提交成功 → 弹出提示"提交成功" → 3秒后自动跳转
（你也可以手动关闭提示窗口）

【修复时间】
大约需要10分钟修改和测试。

【确认】
我现在就改，可以吗？
```

## 场景6：用户对进度不满意

### 问题特征
产品经理觉得开发进度太慢。

### 错误做法
- ❌ 辩解说"技术很复杂"
- ❌ 说"我已经很努力了"
- ❌ 承诺 unrealistic 的时间

### 正确做法
- ✅ 理解用户的焦虑
- ✅ 说明当前进度和已完成内容
- ✅ 解释为什么需要时间（用类比）
- ✅ 提供加速方案（如果有）
- ✅ 重新协商优先级

### 对话示例

**✅ 好的示例**：
```
产品经理：这个功能要这么久吗？

Claude：【理解你的感受】
我理解你希望尽快看到效果，让我说明一下情况。

【当前进度】
已完成：
- ✅ 用户注册（80%）
- ✅ 数据库设计（100%）

进行中：
- 🔄 游戏提交功能（50%）

【时间说明】
游戏提交功能涉及几个部分：
1. 上传文件到服务器（像搬家具到仓库）
2. 保存游戏信息到数据库（像登记入库）
3. 生成游戏列表（像做目录）

这些都需要仔细处理，确保不丢失数据。

【加速方案】
如果时间紧急，我们可以：
1. 先做一个简化版（只保留核心功能）
2. 或者先完成其他更重要的功能

你希望怎么调整？
```

## 场景7：用户提出不切实际的需求

### 问题特征
产品经理提出了技术上不可行或成本过高的需求。

### 错误做法
- ❌ 直接说"做不了"
- ❌ 嘲笑需求不切实际
- ❌ 不解释就拒绝

### 正确做法
- ✅ 理解需求背后的真实目标
- ✅ 说明技术限制
- ✅ 提供替代方案
- ✅ 询问是否愿意降低要求

### 对话示例

**✅ 好的示例**：
```
产品经理：我希望能实时看到每个用户的游戏画面。

Claude：【理解需求】
我理解你想让评审团能实时看到参赛者的游戏画面，
这样评审更直观，对吗？

【技术限制】
这就像要让1000个人同时直播视频，
技术上有几个挑战：
1. 网络带宽成本很高（每月可能需要数万元）
2. 服务器压力很大
3. 用户的电脑配置也要很高

【替代方案】
我建议用这些方案：

方案A：录制视频
- 参赛者录制游戏视频
- 上传到平台
- 评审团观看视频
- 成本低，效果好

方案B：实时截图
- 每隔30秒截一张游戏画面
- 评审团查看截图
- 成本中等

方案C：仅支持部分人实时
- 限制同时在线人数
- 需要排队
- 成本可控

【你的选择】
你觉得哪个方案更符合你的需求？
```

## 通用沟通技巧

### 1. 先确认理解
在采取任何行动前，先复述一遍需求。

### 2. 用类比解释
技术概念用生活场景类比，更容易理解。

### 3. 提供选项
不要给开放式问题，提供具体选项供选择。

### 4. 说明影响
任何决定都会影响时间、成本、效果，要说明清楚。

### 5. 保持诚实
不知道就说不知道，困难就说困难。

### 6. 及时汇报
不要等用户问了才说，主动汇报进度。

## 特殊情况处理

### 用户长时间没回复
- 暂停需要确认的工作
- 继续做已确认的任务
- 保存当前进度和状态
- 在开发日志中记录

### 用户情绪不好
- 先倾听，理解情绪背后的原因
- 不要辩解或争论
- 提供解决方案
- 询问如何帮助

### 需要外部资源
- 提前说明需要什么（设计图、服务器、域名等）
- 说明为什么需要
- 提供获取建议
- 记录在问题清单中
