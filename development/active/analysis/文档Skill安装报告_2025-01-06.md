## 📋 安装的3个文档Skill详解

### 1. docs-write - 文档写作Skill

**来源**: Metabase文档写作风格指南

**核心特点**:
- 对话式、清晰、用户友好的写作风格
- 优先考虑读者理解
- 使用非正式语言,避免术语堆砌

**适用场景**:
- ✅ 编写新文档(设计文档、API文档、用户手册等)
- ✅ 更新现有文档内容
- ✅ 编写技术说明或使用指南

**写作原则**:
1. 匹配复杂度(不过度简化也不过度复杂)
2. 快速给出答案(不拖泥带水)
3. 回答常见问题(基于学习经验)
4. 先告诉做什么,再解释为什么
5. 使用具体的标题(不说废话)

**示例**:
```markdown
✅ 好的写法:
"你可以这样检查游戏评分:进入游戏详情页,点击评分按钮..."

❌ 不好的写法:
"用户应该检查游戏评分。方法是进入游戏详情页,点击评分按钮..."
```

---

### 2. docs-review - 文档审查Skill

**核心功能**:
- 检查文档是否符合写作规范
- 审查PR中的文档变更
- 提供改进建议

**检查项目**:
- **语气和语调**: 是否过于正式、使用"users"而不是"people"
- **结构清晰度**: 重要信息是否被埋没、标题是否具体
- **链接和引用**: 是否使用"点击这里"、链接是否在标题中
- **代码示例**: 代码是否能运行、是否按执行顺序排列
- **格式一致性**: 列表格式、符号使用等

**工作模式**:
1. **PR review模式**: 使用GitHub工具,发布cohesive review
2. **Local review模式**: 输出编号的issue列表

**反馈格式**:
```markdown
**Issue 1: [简要标题]**
行号: 问题描述
建议修复方式
```

---

### 3. doc-coauthoring - 协作文档编写Skill

**核心功能**:
- 引导用户通过结构化流程协作创建文档
- 三阶段工作流程

**三阶段流程**:

**阶段1: 上下文收集**
- 询问文档类型、受众、目标
- 收集相关背景信息
- 询问5-10个澄清问题

**阶段2: 提炼和结构**
- 逐节构建文档
- 头脑风暴5-20个选项
- 用户筛选保留内容
- 迭代完善

**阶段3: 读者测试**
- 用新的Claude(无上下文)测试文档
- 发现盲点
- 确保清晰易懂

**适用场景**:
- ✅ 编写重要的技术规范
- ✅ 创建PRD(产品需求文档)
- ✅ 编写设计文档
- ✅ 创建决策文档
- ✅ 起草RFC(Request for Comments)

## 🔄 如何使用这些Skill

### 场景1: 编写重要的设计文档

**推荐流程**:
```
1. [启动doc-coauthoring workflow]
   你说: "我们使用doc-coauthoring来编写XXX设计文档"
   → 我会问: 文档类型、受众、目标等
   → 收集完上下文后进入下一步

2. [使用docs-write编写内容]
   → 我会按章节逐个编写
   → 遵循对话式风格
   → 提供具体示例

3. [使用docs-review自检]
   → 我会检查语气和格式
   → 验证代码示例
   → 确保清晰易懂

4. [更新相关文档]
   → 同步到CHANGELOG
   → 更新版本号
```

### 场景2: 更新现有文档

**推荐流程**:
```
1. [读取现有文档]
   → 我会了解当前内容
   → 识别需要更新的部分

2. [使用docs-write更新]
   → 修改相关章节
   → 保持风格一致
   → 添加变更说明

3. [使用docs-review检查]
   → 确保更新后文档质量
   → 检查格式一致性

4. [更新版本信息]
   → 更新文档版本号
   → 记录变更内容
```

### 场景3: 快速编写简单文档

**简化流程**:
```
直接使用docs-write
→ 我会自动应用对话式风格
→ 确保清晰易懂
→ 提供具体示例
```
