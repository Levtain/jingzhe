## 🔴 高优先级问题 (9个)

### P1: 7步流程定义不一致

**位置**: 需求文档 301-338行 (提交流程设计)

**问题描述**:
需求中将流程分为"7步",但实际表单分为"7个部分",两者概念混淆:
- 步骤1: 创建游戏→Draft草稿
- 步骤2-7: 对应表单的不同部分

**矛盾点**:
- "7步"是指7个页面跳转?还是单页表单的7个部分?
- 如果是7个页面,用户体验是否过于繁琐?
- 如果是单页表单,为什么要称为"7步"?

**影响**:
- 开发实现时产生歧义
- UI/UX设计方向不明确
- 前端架构选择受影响(多页应用 vs 单页应用)

**建议解决方案**:
```
方案A: 7个页面跳转
  - 优点: 步骤清晰,降低认知负担
  - 缺点: 页面跳转频繁
  - 适用: 复杂表单,用户不熟悉流程

方案B: 单页长表单+锚点导航
  - 优点: 流畅体验,可预览全局
  - 缺点: 页面较长,可能产生视觉疲劳
  - 适用: 熟悉用户,快速填写

推荐: 方案B (单页应用+左侧导航)
```

**决策问题**:
- [ ] 选择方案A还是方案B?
- [ ] 是否需要进度保存提示?
- [ ] 是否支持"上一步/下一步"按钮?

### P2: Draft模式触发时机不明确

**位置**: 需求文档 341-352行 (Draft草稿模式)

**问题描述**:
"创建后自动保存为Draft状态" - 但未说明何时算"创建":
- 填写标题后立即保存?
- 填写标题+简介后保存?
- 点击"创建游戏"按钮时保存?

**潜在场景**:
1. 用户填写标题 → 关闭浏览器 → 是否保存Draft?
2. 用户填写标题+简介 → 不保存离开 → 是否保存Draft?
3. 用户填写所有信息 → 不点击提交 → 是否保留Draft?

**影响**:
- 数据库可能产生大量空Draft记录
- 用户体验:返回时发现很多未完成的草稿
- 存储成本:无用草稿占用空间

**建议解决方案**:
```
Draft保存规则:
1. 自动保存触发条件:
   - 填写标题后3秒自动保存
   - 或点击"保存为草稿"按钮
   - 每30秒自动保存一次(如果有变更)

2. Draft清理规则:
   - 7天未编辑的Draft自动删除
   - 用户可手动删除草稿
   - 提交游戏后自动清除Draft

3. Draft列表管理:
   - "我的草稿"页面显示所有Draft
   - 支持继续编辑/删除
   - 显示最后编辑时间
```

**决策问题**:
- [ ] 何时触发首次Draft保存?
- [ ] Draft自动清理周期是多少天?
- [ ] 是否需要Draft数量限制(如最多3个)?

### P4: 字符限制单位不统一

**位置**: 多处字段定义

**问题描述**:
需求中字符限制单位不统一:
- 游戏标题: "5-100字符"
- 游戏简介: "确切100字"
- 游戏正文: "最少200字符"
- 链接描述: "最小字符数: 10字符"

**严重矛盾**:
- "100字"是中文字符?英文单词?字节?
- "100字符"是ASCII字符?UTF-8字符?
- 中文环境下"1个中文字=2个字符"?还是"1个中文字=1个字符"?

**影响**:
- 用户填写时无法准确判断是否符合要求
- 前后端验证规则可能不一致
- 多语言支持时出现问题

**建议解决方案**:
```
统一使用"字符"作为单位:
1. 明确字符计数规则:
   - 1个中文字 = 1个字符
   - 1个英文字母 = 1个字符
   - 1个Emoji = 2个字符(需特殊说明)
   - 1个标点符号 = 1个字符

2. 统一需求描述:
   - 游戏标题: 5-100个字符
   - 游戏简介: 恰好100个字符
   - 游戏正文: 至少200个字符
   - 链接描述: 至少10个字符

3. 前端实时计数:
   - 显示"当前: 87/100 字符"
   - 不足时提示"还需13个字符"
   - 超出时提示"超出3个字符"
```

**决策问题**:
- [ ] 统一使用"字符"还是"字"作为单位?
- [ ] Emoji如何计数?(1个还是2个字符?)
- [ ] 是否需要前端实时计数器?

- 少于100字怎么办?(提示补充?拒绝提交?)

- 英文单词如何计数?

```
3. UI交互设计:
   [游戏简介] (必须恰好100字)
   ┌────────────────────────┐
   │ 在这个星球上,玩家需要...│ 87/100
   │                        │
   └────────────────────────┘
   ⚠️ 请再补充13个字符

   [显示字数统计] [智能补全建议]
```

**决策问题**:

- [ ] 是否提供智能补全建议?(如"建议补充...")
- [ ] 少于100字时是否允许提交?(如95-105字范围?)

### P7: 游戏链接"有效性检查"未定义

**位置**: 需求文档 152-162行 (2.4 链接与描述)

**问题描述**:
"链接有效性检查" - 但未说明:
- 前端检查什么?(URL格式?)
- 后端检查什么?(HTTP状态码?)
- 同步检查还是异步检查?
- 超时时间是多少?
- 404/403如何处理?

**技术挑战**:
- 百度网盘等网盘链接需要登录才能访问,如何验证有效性?
- Google Drive的分享链接可能需要权限,如何处理?
- itch.io的链接可能是私有项目,如何检测?

**影响**:
- 用户提交无效链接,评分者无法下载游戏
- 后端验证可能超时,影响用户体验
- 特殊平台链接验证逻辑复杂,开发成本高

**建议解决方案**:
```
链接验证策略:
1. 前端验证 (即时):
   - URL格式正则检查
   - 常见域名白名单检查
   - 危险协议拦截 (javascript:, data:)

2. 后端验证 (异步):
   - HEAD请求检查HTTP状态码
   - 超时时间: 5秒
   - 允许的HTTP状态: 200, 301, 302
   - 失败时标记为"需验证"但不阻止提交

3. 特殊平台处理:
   - 百度网盘: 仅验证URL格式,提示用户确保链接正确
   - Google Drive: 尝试请求,失败标记为"需验证"
   - itch.io: 检查项目是否存在
   - 私有服务器: HEAD请求验证

4. UI反馈:
   [链接URL] https://pan.baidu.com/s/xxx
   ✅ 链接格式正确
   ⏳ 正在验证可访问性...
   ⚠️ 无法验证链接有效性(网盘链接),请确保链接正确
```

**决策问题**:
- [ ] 后端验证超时时间设置为多少秒?(建议5秒)
- [ ] 验证失败时是否阻止提交?(建议不阻止,仅警告)
- [ ] 是否定期重新验证已提交的链接?(建议每天一次)

### P15: 编辑器技术选型成本被低估

**位置**: 需求文档 39-54行 (1.3 游戏正文描述) + 问题清单93-129行

**问题描述**:
需求要求"Markdown编辑器+实时预览",确认使用CodeMirror 6或Monaco Editor,但实现复杂度被低估:
- 需要配置Markdown语法高亮
- 需要实现实时预览同步滚动
- 需要集成图片上传功能
- 需要实现工具栏快捷插入
- 移动端适配问题

**技术挑战**:
1. CodeMirror 6集成成本:
   - 学习曲线陡峭(新API与传统版本差异大)
   - 配置复杂,需要大量自定义
   - 文件大小约200KB (gzipped)

2. 移动端适配:
   - 手机屏幕小,左右分屏体验差
   - 是否需要切换按钮(编辑/预览)?
   - 触摸屏是否支持滑动切换?

3. 图片上传与Markdown语法:
   - 上传后自动插入 `![alt](url)` 语法
   - 需要处理图片URL拼接
   - 需要处理图片大小限制

**影响**:
- 开发周期可能延长5-7天
- 技术风险:团队成员不熟悉CodeMirror 6
- 移动端体验可能不佳

**建议解决方案**:
```
技术选型对比:
方案A: CodeMirror 6 (功能完整)
  - 优点: 功能强大,可定制性高
  - 缺点: 学习曲线陡峭,开发周期长
  - 开发时间: 约5-7天
  - 文件大小: 约200KB (gzipped)

方案B: Monaco Editor (VS Code同款)
  - 优点: 功能强大,微软维护
  - 缺点: 重量级,约2MB
  - 开发时间: 约5-7天
  - 文件大小: 约2MB (gzipped后约500KB)

方案C: 简化版 (纯文本域+实时预览)
  - 优点: 开发快,轻量
  - 缺点: 功能有限,无语法高亮
  - 开发时间: 约2-3天
  - 文件大小: 约50KB

方案D: 现成库 (推荐第一届)
  - toast-ui.editor (韩国,功能完整)
  - mavon-editor (中文友好,Vue生态)
  - milkdown (现代,基于ProseMirror)
  - 优点: 开箱即用,文档丰富
  - 缺点: 定制性较低
  - 开发时间: 约1-2天

第一届建议: 方案D (toast-ui.editor或mavon-editor)
理由: 快速上线,降低技术风险,后续可迭代优化
```

**决策问题**:
- [ ] 选择哪个技术方案?(建议D)
- [ ] 如果选择D,选择toast-ui.editor还是mavon.editor?
- [ ] 移动端是否提供简化版?(切换按钮而非分屏)

**决策问题**:

### P9: 评分期版本锁定机制冲突

**位置**: 需求文档 397-415行 (更新与版本管理)

**问题描述**:
"评分期开始:不允许更新(版本锁定)"
但评分系统文档中说"评分修改规则:允许修改直到评分阶段结束"

**潜在冲突**:
- 评分期开始后游戏不能更新
- 但评分者可以修改评分
- 如果游戏发现严重bug需要紧急修复,怎么办?

**边界场景**:
1. 游戏提交后发现严重bug(无法启动/崩溃)
   - 开发期内可以更新
   - 评分期开始后无法更新
   - 但如果评分期才1天,bug修复后无法更新是否合理?

2. 游戏链接失效(网盘链接过期)
   - 评分期内无法更新链接
   - 评分者无法下载游戏评分
   - 是否应该允许更新链接?

**影响**:
- 严重bug导致游戏无法体验,影响评分公平性
- 链接失效导致评分者无法评分
- 开发者无法修正问题,用户体验差

**建议解决方案**:
```
版本锁定规则细化:
1. 锁定范围:
   - 游戏描述、截图、视频等营销内容 → 完全锁定
   - 游戏下载链接 → 允许修正(需管理员审核)
   - 严重Bug修复 → 允许紧急更新(需管理员审批)

2. 紧急更新流程:
   开发者提交申请:
   - 说明问题类型(链接失效/严重bug)
   - 提供修正方案(新链接/补丁说明)
   - 提交申请

   管理员审核:
   - 确认是严重bug或链接失效
   - 评估是否允许更新
   - 批准后允许更新

   更新后处理:
   - 更新历史中标记"紧急更新"
   - 已评分的用户可选择是否重新评分
   - 通知所有已评分用户

3. 链接失效处理:
   - 用户举报链接失效
   - 系统自动邮件通知开发者
   - 开发者24小时内修正链接
   - 超时未修正: 游戏隐藏+标记"链接失效"
```

**决策问题**:
- [ ] 是否允许评分期紧急更新?(建议允许,但需审核)
- [ ] 哪些情况算"严重bug"?(建议:无法启动/卡死/数据丢失)
- [ ] 链接失效是否允许修正?(建议允许,24小时内)

### P11: Draft草稿的重复提交防护缺失

**位置**: 需求文档 341-352行 (Draft草稿模式)

**问题描述**:
需求中提到"同一团队不能重复提交"(377行),但Draft模式下如何防止重复提交?

**边界场景**:
1. 用户创建Draft1 → 填写一半 → 放弃 → 创建Draft2 → 提交
   - Draft1是否保留?是否需要清理?

2. 用户创建Draft → 提交 → 又想修改 → 撤回 → 创建新Draft
   - 如何区分"正常撤回修改"和"重复提交"?

3. 多个队长成员同时操作:
   - 队长A创建Draft
   - 副队长B也创建Draft
   - 两人都提交 → 谁的提交有效?

**影响**:
- 数据库可能产生大量无用Draft
- 多人同时操作导致数据冲突
- 用户可能重复提交同一游戏

**建议解决方案**:
```
Draft与重复提交防护:
1. Draft唯一性限制:
   - 每个团队同时只能有1个活跃Draft
   - 创建新Draft前,必须删除或发布旧Draft
   - UI提示: "您已有未完成的草稿,是否继续?"

2. 多人协作冲突:
   - 队长A正在编辑Draft
   - 副队长B尝试编辑 → 提示"队长A正在编辑,建议等待"
   - 使用乐观锁:记录最后编辑时间和编辑者
   - 自动保存时检测冲突,提示用户

3. 重复提交检测:
   - 提交前检查:该团队是否已有已发布的游戏
   - 如果有 → 弹窗确认:
     """
     您的团队已提交游戏《XXX》
     是否确定提交新游戏?旧游戏将保留但不可编辑。

     [取消] [确认提交]
     """

4. Draft清理机制:
   - 提交游戏后自动清除Draft
   - 7天未编辑的Draft自动删除
   - 用户可手动删除Draft
```

**决策问题**:
- [ ] 每个团队是否限制只能有1个活跃Draft?(建议是)
- [ ] 多人同时编辑时如何处理?(建议锁定机制)
- [ ] 已提交游戏后是否允许重新提交?(建议允许撤回后重新提交)

### P12: 游戏撤回后的作者列表锁定问题

**位置**: 问题清单236-243行 (Q6:提交撤回机制)

**问题描述**:
"允许在评分期前撤回"
"修改后重新提交"
但撤回后作者列表是否仍然锁定?

**场景**:
1. 3人团队提交游戏 → 撤回 → 1个成员退出 → 重新提交
   - 新作者列表是3人还是2人?
   - 退出的成员是否还显示为作者?

2. 3人团队提交游戏 → 撤回 → 添加2个新成员 → 重新提交
   - 新作者列表是3人还是5人?
   - 新成员是否显示为作者?

**与团队系统的冲突**:
- 团队系统文档说"提交时记录快照"
- 如果撤回后重新提交,是否更新快照?

**影响**:
- 数据一致性:作者快照与当前团队成员不一致
- 公平性:未参与创作的成员被列为作者
- 或参与创作的成员未被列为作者

**建议解决方案**:
```
撤回后的作者列表处理:
选项A: 撤回后解锁,重新提交时更新快照
  优点: 灵活,支持成员变更
  缺点: 可能被滥用(频繁换人提交)
  适用: 允许团队动态调整

选项B: 撤回后仍锁定,必须删除游戏才能重新提交
  优点: 简单,防止滥用
  缺点: 不够灵活,不支持成员变更
  适用: 强调团队稳定性

选项C: 撤回后锁定,但允许"删除并重新创建"
  - 需要管理员审核
  - 删除历史中记录原作者
  - 重新创建时更新作者快照
  优点: 平衡灵活性和防滥用
  缺点: 流程复杂,需管理员介入

推荐: 选项A (撤回后解锁,但记录修改历史)

实现细节:
1. 撤回时:
   - 游戏状态: published → withdrawn
   - 作者快照: 保留历史记录
   - 版本号: 递增(v1 → v2)

2. 重新提交时:
   - 检查当前团队成员
   - 更新作者快照
   - 记录修改历史:
     - v1: 作者[张三, 李四, 王五]
     - v2: 作者[张三, 李四, 赵六] (王五退出,赵六加入)

3. 版本对比:
   - 游戏详情页显示作者变更历史
   - 标注"作者列表已变更"
```

**决策问题**:
- [ ] 选择选项A、B还是C?(建议A)
- [ ] 如果选择A,是否限制撤回次数?(建议最多3次)
- [ ] 是否显示作者变更历史?(建议显示)

### P16: 数据库外键约束缺失

**位置**: 需求文档 484-549行 (数据表设计建议)

**问题描述**:
定义了多个数据表,但没有说明外键约束:
- games表关联team_id,但没有说明:
  - 如果team被删除,游戏数据如何处理?
  - 是否使用ON DELETE CASCADE?
  - 是否使用软删除?

**潜在数据一致性问题**:
1. 团队解散 → team表删除记录
   - games表的team_id变为孤儿记录
   - 如何查询游戏对应的团队?

2. 成员退出 → team_members表删除记录
   - 但game_authors快照表仍保留
   - 用户信息变更(改名)后,快照是否更新?

**影响**:
- 数据完整性问题:孤儿记录
- 查询复杂度增加:需要LEFT JOIN
- 数据一致性问题:团队删除但游戏仍存在

**建议解决方案**:
```
外键约束设计:
1. teams表 (团队主表)
   - 使用软删除: deleted_at DATETIME NULL
   - 不物理删除团队记录
   - 查询时: WHERE deleted_at IS NULL

2. games表 (游戏主表)
   team_id INT NOT NULL,
   FOREIGN KEY (team_id)
     REFERENCES teams(id)
     ON DELETE RESTRICT,  -- 禁止删除有游戏的团队
     ON UPDATE CASCADE

   说明: 如果团队有已发布的游戏,禁止删除团队
         必须先删除或隐藏游戏,才能删除团队

3. game_media表 (媒体资源表)
   game_id INT NOT NULL,
   FOREIGN KEY (game_id)
     REFERENCES games(id)
     ON DELETE CASCADE,  -- 删除游戏时级联删除媒体
     ON UPDATE CASCADE

   说明: 删除游戏时,自动删除关联的封面图、截图等

4. game_authors表 (作者快照表)
   - 不设置外键到team_members表
   - 独立快照数据
   - 即使team删除,作者快照仍保留
   - 记录snapshot_at快照时间

5. 软删除策略:
   所有表添加:
   - deleted_at DATETIME NULL
   - 查询时: WHERE deleted_at IS NULL
   - 删除时: UPDATE SET deleted_at = NOW()

6. 数据一致性检查:
   定期任务(每天一次):
   - 检查孤儿记录
   - 修复或标记孤儿记录
   - 发送警告给管理员
```

**决策问题**:
- [ ] 使用软删除还是物理删除?(建议软删除)
- [ ] team删除时是否禁止(有游戏的团队)?(建议禁止)
- [ ] game_authors是否设置外键?(建议不设置,独立快照)
