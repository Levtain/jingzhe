## 1. 项目概述

### 1.1 定义

惊蛰计划是一项限时游戏创作活动，要求参与者在主题公布后 48 小时内完成一款可游玩的游戏 Demo。

特点：

- 主题由全体用户参与征集与投票
- 评分由参赛者与评审团共同完成
- 设有微型经济系统（金币与打赏）
- 提倡互动互评与社区氛围
- 同时支持个人赛道与团队赛道

### 1.2 系统目标

- 支撑完整的活动全流程运营
- 确保规则执行自动化
- 保证公平、公正、透明
- 降低人工干预成本
- 提供风控与异常治理能力

## 2. 角色与身份体系

### 2.1 身份类型

| 身份 | 获得方式 | 权限 |
|------|----------|------|
| 路人玩家 | 注册但未提交 | 试玩、评论、打赏、推荐 |
| 参赛者 | 开发期内提交游戏 | 同路人 + 评分 |
| 评审团 | 官方邀请 | 评分权重加倍、推荐特权 |

### 2.2 身份判定优先级

评审团 > 参赛者 > 路人玩家

### 2.3 自动判断逻辑

```
IF 标记为评审团 -> 评审团
ELSE IF 提交有效作品 -> 参赛者
ELSE -> 路人玩家
```

### 2.4 组队与赛道

- 两种赛道：个人赛 / 团队赛  
- 团队上限人数：9 人  
- 同一用户每次活动仅参与一款游戏  
- 团队成员需在赛程结束前绑定，否则：
  - 不享受作者身份
  - 不享受团队奖励和收益

## 3. 主题征集与投票流程

> **核心价值**：公平、公开、好玩

### 3.0 完整流程

| 阶段 | 时间 | 参与人群 | 说明 |
|------|------|-----------|------|
| 主题征集 | T-4周，7天 | 所有注册用户 | 每人提交1个主题 |
| 主题筛选 | T-3周，7天 | 所有注册用户 | 多轮随机抽样投票 |
| 三轮投票 | T-2周到T-1周 | 所有注册用户 | R1、R2、R3各3天 |
| 最终抽签 | T日之前 | 主办方 | 直播抽签决定最终主题 |

---

### 3.1 主题征集阶段（T-4周，7天）

**参与规则**：
- 所有注册用户均可参与
- 每人只能提交1个主题
- 主题要求：简洁、可创作、不限形式

**停止条件**（满足其一即可）：
- 未去重前主题数量超过1200个
- 征集时间达到7天

**兜底机制**：
- 最小征集量：100个（去重后）
- 如果不足100个，由主办方从已征集的主题中随机选择补充

---

### 3.2 主题筛选阶段（T-3周，7天）

**核心机制**：多轮随机抽样投票（点赞冒泡）

#### 3.2.1 基础规则

**轮次自适应**：
- 筛选池 ≤ 200个：3轮
- 筛选池 200-400个：6轮
- 筛选池 ≥ 400个：9轮

**每轮规则**：
- 随机抽取10个不重复的主题
- 每个用户最多点赞6次
- 每个用户在每轮抽到的主题各不相同
- 使用混合方案+分层抽样算法，保证每个主题都被抽到

**激励措施**：
- 每日投出总主题量1/3的赞 → 获得5金币
- 成就系统（待后续设计）

#### 3.2.2 抽样算法

**分层抽样**：
- 将主题池平均分成N组（N=轮次数）
- 每轮从不同组中抽取
- 保证每个主题至少被抽到1次

**混合调度**：
- 前N-2轮：完全随机抽样（保持游戏性）
- 后2轮：优先抽取从未出现或出现次数<2的主题（保证完整性）

**用户差异化**：
- 为每个用户使用不同的随机种子
- 确保每个人看到的主题都不相同

#### 3.2.3 筛选结果

**统计方式**：
- 统计所有轮次的点赞总数
- 按点赞数从高到低排序
- 取点赞数最高的28个主题进入三轮投票

---

### 3.3 三轮投票阶段（T-2周到T-1周）

#### 3.3.1 第一轮投票（R1，3天）

**投票池**：28个主题（筛选阶段点赞量最高的）

**随机抽取**：从中抽取12个主题

**投票规则**：
- 每个用户4张票
- 每个主题最多投1票
- 可以少投，也可以不投

**结果处理**：
- 统计票数
- 票数最高的4个进入第二轮
- 平票处理：从平票的选项中随机抽取1个

**票数显示**：
- 投票过程中不显示实时票数
- 投票结束后显示本轮结果

---

#### 3.3.2 第二轮投票（R2，3天）

**投票池**：剩余16个主题

**随机抽取**：从中抽取12个主题

**投票规则**：
- 每个用户4张票（规则同R1）

**结果处理**：
- 统计票数
- 票数最高的4个进入第三轮
- 平票处理：从平票的选项中随机抽取1个

**未被抽取的主题**：4个（直接进入第三轮）

**票数显示**：
- 投票结束后显示本轮结果

---

#### 3.3.3 第三轮投票（R3，3天）

**投票池组成**：
- R1选出的4个
- R2选出的4个
- R2未被抽取的4个
- 合计：12个主题

**投票规则**：
- 每个用户4张票（规则同R1）

**结果处理**：
- 统计票数
- 票数最高的4个进入最终主题池
- 平票处理：从平票的选项中随机抽取1个

**票数显示**：
- 投票结束后显示本轮结果

---

### 3.4 最终主题选择

**抽签方式**：主办方从4个主题中直播抽签

**公布时间**：惊蛰后第一个周五 17:00

**仪式感**：建议做成公开活动，增加参与感

---

### 3.5 反作弊措施

- 账号年龄限制
- IP 与设备指纹识别
- 投票速率限制
- 异常票权重自动降低或移除
- 举报与人工复核机制

---

### 3.6 核心设计理念

**公平**：
- 每个主题都有机会被看到
- 随机抽样，无人为干预
- 算法公开透明

**公开**：
- 规则清晰明确
- 结果公开可见
- 过程可追溯

**好玩**：
- 盲盒体验（随机抽取）
- 成就系统（金币激励）
- 游戏化投票（而非枯燥的列表）

## 4. 活动流程与时间线

```
惊蛰后第一个周五 17:00 公布主题
→ 48小时开发期
→ 12天互玩互评
→ 结果公布
```

支持：

- 自动状态流转

- 宕机或不可抗力顺延

## 5. 游戏开发与提交规则（开发准则对齐）

### 5.1 必须可游玩

必须包含：

- 可启动执行文件
- 所需运行库
- 无需额外复杂安装步骤

### 5.2 不建议但允许

- 预制项目结构
- 常用代码与工具包

### 5.3 素材规则

允许：

- 自制资源
- 公有领域素材
- 已购买授权素材

必须：

- 标注来源
- 明确授权

### 5.4 生成式 AI 使用

- 不建议使用 AI 制作美术/音频
- 如使用必须标注
- 自动限制相关评分维度参与

### 5.5 提交后更新

允许：Bug 修复  
禁止：功能新增  

系统支持：

- 版本号记录
- 更新日志
- 高亮变更内容

### 5.6 内容适宜性

禁止：

- 违法违规
- 过度暴力、色情
- 违背公序良俗

违规则：取消评选资格或下架

## 6. 评分系统设计

### 6.1 评分维度

- 创新性  
- 主题诠释  
- 视觉效果  
- 音乐音频  
- 整体性  
- 整活儿（娱乐性/完成度综合向）  

### 6.2 评分范围

- 0–10 整数档
- 参赛者与评审团可评分
- 路人玩家不可评分

### 6.3 权重

- 评审团评分计入总评分人次时 ×2
- 计算公式与权重参数保留后续讨论

### 6.4 特殊处理

- 使用 AI 或第三方素材 → 不参与视觉/音频维度评分  
- 自评允许性：可配置  
- 最低评分数量要求：可配置  

---

## 7. 排名与结果公布

### 7.1 排名资格

总评分人次 > 20  
其中：

```
总评分人次 = 参赛者评分人次 + 2 × 评审团评分人次
```

### 7.2 三组结果独立公布

- 参赛者评分榜
- 评审团评分榜
- 赏金收益情况展示（不排名）

### 7.3 并列处理

优先级：

1. 评审团得分
2. 有效评分人数
3. 创新性维度得分

## 8. 经济系统（规则落地，不含数值模型）

### 8.1 金币属性

- 非法币
- 不支持提现
- 可购买站内服务（如推荐位）

### 8.2 核心规则

- 评分后必须打赏游戏（1–10 金币）
- 评论打赏 0–5 金币
- 参赛者金币每日清零
- 评审团金币不清零
- 赛后全部金币清零
- 心理价位由作者设定并且赛后公开

### 8.3 金币来源与结算

- 身份基础金币
- 评分、活跃额外金币
- 赛后获得已收到赏金等额金币

## 9. 推荐位与推荐机制

### 9.1 基本规则

- 推荐位总数：12 个
- 其中系统保留部分用于“评分人次不足20”的作品
- 其余支持金币购买

### 9.2 展示规则

- 高峰时间段优先展示购买推荐位
- 非高峰优先展示“危险边缘作品”
- 单次展示约 30–45 分钟
- 同游戏每日仅一次

## 10. 个人赛与团队赛

### 10.1 个人赛

要求：

- 独立完成开发与互动
- 素材 48 小时内完成（例外条款除外）

可得额外奖励：

- 公布源码 → 额外金币
- 免费推荐位机会若干

### 10.2 团队赛

规则：

- ≤ 9 人
- 所有成员必须绑定项目
- 开发权限共享
- 素材规则同开发准则

## 11. 善意度、反作弊与风控机制

### 11.1 善意度来源

- 评分次数
- 有效评论被点赞
- 正常活跃行为

### 11.2 善意度影响

- 首页曝光提升
- 超阈值后反向衰减以防刷票
- 极端情况下影响排名权重（可配置）

### 11.3 反作弊范围

- 刷票
- 互刷评分
- 机器人注册
- 赏金洗钱
- 评审团滥用权力

### 11.4 识别技术

- 设备指纹
- 行为轨迹聚类
- 异常评分模式识别
- IP 相关性检测

## 16. 体系公平性评估（新增结论性说明）

总体判断：**相对公平，但非完全客观**。

优势：

- 双榜制度兼顾大众与专业
- 多维度评分降低情绪偏置
- 评分人数门槛减少小样本波动
- AI 使用限制保护人工创作

风险：

- 互评天然存在社交偏置
- 打赏与曝光正反馈导致马太效应
- 善意度黑箱易造成质疑
- 参赛者视角 ≠ 普通消费者视角

因此：

- 更适合作为“比赛规则”
- 不完全适合作为“客观市场评价体系”

## 17. 市场评价体系可行性分析（新增）

当前制度：

- 更偏向创作竞赛
- 发现潜力与交流价值大
- 市场客观性有限

不能直接成为：

- 行业定价标准
- 商业化成功预测工具

但可以成为：

- 创作者社群权威榜单
- 潜力作品发现平台
- 行业内口碑参考坐标系

## 18. 建议与未来发力方向（新增核心内容）

如果目标是建立：

> 可靠的市场级“评价与定价参考体系”

需要重点在以下方向发力：

### 18.1 评价体系科学化

- 从主观互评分升级为“多源融合评价”  
- 融合：
  - 参赛者评分
  - 评审团意见
  - 路人玩家评分
  - 第三方长期玩家反馈

- 引入统计学：
  - 置信区间
  - 极端值处理
  - 样本量加权

### 18.2 行为数据引入

除主观评价外增加：

- 游戏时长
- 重复游玩率
- 留存（Day1 / Day7）
- 推荐意愿（NPS）
- 自愿付费行为

→ 用行为信号修正主观打分

### 18.3 定价参考体系建设

- 建立“内容量–质量–定价”映射
- 引入以下变量：
  - 玩法深度
  - 市场定位
  - 目标人群规模
  - 竞品价格带
  - 开发成本与团队规模

目标：

- 提供“建议定价区间”  
- 建立“游戏价值指数”
