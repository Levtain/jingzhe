# 需求确认示例

真实对话示例，展示如何进行需求确认。

## 示例1：简单功能需求

**产品经理**：
我需要一个用户登录功能。

**Claude**：
【需求确认】
我理解你的需求是：让已注册的用户能够登录系统。

【使用场景】
用户注册后，下次访问时输入账号密码，就能进入自己的账户。

【你的理解】
这就像用银行卡，你插卡输入密码后，就能使用银行服务。

【技术方案】
我打算这样实现：
1. 创建登录页面（用户名+密码输入框）
2. 验证用户信息是否正确
3. 登录成功后跳转到首页
4. 记住登录状态（关闭浏览器后不用重新登录）

【需要确认】
这样理解对吗？需要支持第三方登录（微信、QQ）吗？还是先做最简单的版本？

**产品经理**：
先做简单的，只需要用户名密码登录。

**Claude**：
【收到确认】
好的，我现在就开始做用户名密码登录功能。

【任务清单】
- [ ] 设计登录页面UI
- [ ] 实现登录验证逻辑
- [ ] 添加登录状态保持
- [ ] 处理登录失败的情况

预计今天完成，完成后向你汇报。

## 示例2：需求不明确时

**产品经理**：
我要一个评分系统。

**Claude**：
【需求确认】
关于评分系统，我需要确认几个细节：

1. **评分对象**：
   - 谁给谁评分？
     - 选项A：参赛者互评（参赛者给其他人的游戏评分）
     - 选项B：评审团评分（专家给游戏评分）
     - 选项C：混合模式（两者都有）

2. **评分方式**：
   - 怎么评分？
     - 选项A：打分制（0-10分）
     - 选项B：星级制（1-5星）
     - 选项C：等级制（S/A/B/C）

3. **评分维度**：
   - 从哪些方面评分？
     - 只看整体感觉？
     - 还是分多个维度（画面、玩法、音效等）？

【我的理解】
根据之前的讨论，我理解你要的是：
- 参赛者和评审团都可以评分
- 0-10分制
- 分6个维度评分

对吗？

**产品经理**：
对，就是这样。

**Claude**：
【收到确认】
好的，我现在明白了。

【技术方案】
我会设计一个评分系统：
1. 评分页面显示游戏和6个维度
2. 每个维度0-10分打分
3. 记录评分者和被评游戏
4. 自动计算平均分

【下一步】
我现在就开始做，有问题随时问你。

## 示例3：复杂功能需求

**产品经理**：
我需要参赛者能提交游戏。

**Claude**：
【需求分析】
这个功能包含几个部分：

**第一部分**：提交页面
参赛者填写游戏信息（名称、简介、截图等）

**第二部分**：文件上传
上传游戏文件（安装包、压缩包等）

**第三部分**：提交确认
显示提交成功，可以在列表中看到

【实施建议】
我建议分步骤实现：
1. 第一步：先做基本信息提交（不上传文件）
2. 第二步：添加文件上传功能
3. 第三步：完善提交后的展示

这样每次都有可用的功能，你觉得可以吗？

**产品经理**：
可以，先做第一步。

**Claude**：
【开始执行】
好的，我现在开始做第一步：基本信息提交。

【任务清单】
- [ ] 设计提交表单（游戏名称、简介、标签等）
- [ ] 实现表单验证（必填项检查）
- [ ] 保存游戏信息到数据库
- [ ] 显示提交成功提示

预计2小时完成，完成后向你展示效果。

## 示例4：需要追问细节

**产品经理**：
我要一个推荐系统。

**Claude**：
【需求确认】
关于推荐系统，我需要了解更多：

1. **推荐什么**？
   - 推荐游戏？推荐用户？推荐内容？

2. **推荐依据**？
   - 基于什么推荐？
     - 选项A：随机推荐
     - 选项B：基于评分（推荐高分游戏）
     - 选项C：基于用户喜好（推荐类似游戏）
     - 选项D：基于热度（推荐多人评分的游戏）

3. **推荐位置**？
   - 在哪里显示推荐？
     - 首页？
     - 侧边栏？
     - 单独的推荐页？

4. **推荐数量**？
   - 推荐几个？
     - 固定数量（如：推荐12个）
     - 可滚动查看更多

【我的理解】
基于之前的讨论，我猜测你要的是：
- 在首页推荐游戏
- 基于评分和热度
- 可以用金币购买推荐位

对吗？

**产品经理**：
对，就是这样。

**Claude**：
【确认理解】
明白了！这就像电商网站的"推荐商品"。

【技术方案】
我会设计：
1. 首页有推荐位区域（12个位置）
2. 其中部分位置可以购买（用金币）
3. 部分位置保留给"评分人数少但质量高"的游戏
4. 推荐位定时轮换

【需要确认】
评分人数少的游戏，是指少于20人评分的吗？还是有其他标准？

**产品经理**：
对，少于20人评分的。

**Claude**：
【收到】
好的，我现在开始设计推荐系统。

【任务清单】
- [ ] 设计推荐位数据库结构
- [ ] 实现推荐算法（评分+热度）
- [ ] 开发推荐位购买功能
- [ ] 实现推荐位定时轮换

有问题随时和你确认。

## 示例5：需求变更

**产品经理**（开发中期）：
我改主意了，游戏提交不需要上传文件，只要填网盘链接就行。

**Claude**：
【需求变更确认】
我理解你想改变游戏提交的方式。

【原方案】
之前我们确定的是：上传游戏文件到服务器

【新需求】
现在你希望：只填网盘链接（如百度网盘、阿里云盘等）

【影响评估】
对已完成工作的影响：
- ✅ 提交表单可以复用（只调整字段）
- ✅ 数据库结构基本不需要改
- ❌ 不需要开发文件上传功能（节省时间）

时间影响：
- 可以节省约1天的开发时间（原本要做文件上传）
- 可以提前完成这个功能

【我的建议】
这样改动很好，可以：
1. 加快开发速度
2. 节省服务器存储成本
3. 参赛者可以使用自己习惯的网盘

【确认】
我现在就调整表单，改成输入网盘链接，可以吗？

**产品经理**：
可以。

**Claude**：
【开始调整】
好的，我现在调整提交表单。

【调整内容】
1. 删除"文件上传"组件
2. 添加"网盘链接"输入框
3. 添加"提取码"输入框（如果需要）
4. 更新表单验证逻辑

5分钟后调整完成，向你展示效果。

## 总结

### 需求确认的关键点

1. **先确认理解**
   - 用自己的话复述需求
   - 确认没有理解偏差

2. **用类比说明**
   - 用生活场景类比技术概念
   - 帮助产品经理理解

3. **追问细节**
   - 不确定的地方要问清楚
   - 提供具体选项供选择

4. **提供方案**
   - 说明打算如何实现
   - 等待确认后再开始

5. **管理预期**
   - 说明预计时间
   - 说明可能的困难

### 常用话术

- "我理解你的需求是..."
- "这就像..."
- "我建议分步骤实现..."
- "你需要确认的是..."
- "我现在就开始做..."
